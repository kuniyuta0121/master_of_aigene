----------------------------------------------------------------------
  1.1 確率分布 — データの「形」を捉える道具
----------------------------------------------------------------------
  まず全体像:
  データサイエンスは「値そのもの」より「値がどう出現するか」を扱う学問。
  この出現パターンを表現する言語が確率分布。
  
  確率分布とは:
  - 何が起きうるか（取りうる値の範囲）
  - どれくらい起きやすいか（確率）
  を定義したモデル。
  
  なぜ重要か:
  - ノイズの多い観測値から全体傾向を推定できる
  - まだ見ていないデータへの予測ができる
  - 検定・推定・機械学習の前提を明示できる
  
  アナロジー:
  公平な6面サイコロは1回ごとの結果は読めないが、
  長期では各目がほぼ1/6に収束する。この「短期の不確実性と長期の規則性」
  を同時に扱えるのが分布の強み。

----------------------------------------------------------------------
  主要な分布一覧
----------------------------------------------------------------------
  まず全体像: この節では「主要な分布一覧」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  ┌──────────────┬───────────────────┬──────────────────┐
  │ 分布         │ 用途              │ 例               │
  ├──────────────┼───────────────────┼──────────────────┤
  │ 正規分布     │ 自然現象の近似    │ 身長・テスト点数 │
  │ ベルヌーイ   │ 成功/失敗の1回    │ コイン投げ       │
  │ 二項分布     │ 成功/失敗のn回    │ CTR (n回表示)    │
  │ ポアソン分布 │ 単位時間の回数    │ 1時間のアクセス数│
  │ 指数分布     │ 次のイベントまで  │ 故障までの時間   │
  │ 一様分布     │ 等確率            │ 乱数生成         │
  └──────────────┴───────────────────┴──────────────────┘
  
  正規分布の核心: 中心極限定理 (CLT)
  → どんな分布でも、十分な数の平均を取ると正規分布に近づく。
  → だから「平均値」を扱うとき正規分布が使える。

----------------------------------------------------------------------
  中心極限定理デモ (一様分布 → 平均の分布)
----------------------------------------------------------------------
  一様分布 U(0,1) から30個サンプルし平均 × 1000回
  平均の平均: 0.4998 (理論値 0.5)
  平均の標準偏差: 0.0523 (理論値 0.0527)

  ⚠ よくある誤解: 正規分布はすべてのデータに当てはまる
  → 正確には: 収入・SNSフォロワー数などは対数正規分布やべき乗則に従う

----------------------------------------------------------------------
  1.2 ベイズ定理 — 「証拠」で信念を更新する
----------------------------------------------------------------------
  この節の狙い: 「1.2 ベイズ定理 — 「証拠」で信念を更新する」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  P(A|B) = P(B|A) × P(A) / P(B)
  
  事前確率 P(A): 証拠を見る前の信念
  尤度 P(B|A): Aが真のときBが観測される確率
  事後確率 P(A|B): 証拠Bを見た後の信念
  
  ■ アナロジー: 医療検査
  病気の有病率が0.1% (事前確率が低い) の場合、
  検査陽性でも実際に病気である確率は意外と低い。

  有病率: 0.1%
  感度(真陽性率): 99%
  特異度(真陰性率): 95%
  検査陽性時の実際の罹患確率: 1.9%
  → 陽性でも約98%は偽陽性！事前確率が低いと起こる現象

  ⚠ よくある誤解: ベイズ統計はフリクエンティスト統計より優れている
  → 正確には: 問題の性質による。事前知識が活かせる場面でベイズが強い

----------------------------------------------------------------------
  1.3 仮説検定 — 「偶然か否か」を判定する
----------------------------------------------------------------------
  この節の狙い: 「1.3 仮説検定 — 「偶然か否か」を判定する」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  手順:
  1. 帰無仮説 H₀ を立てる (差がない / 効果がない)
  2. 対立仮説 H₁ を立てる (差がある)
  3. 検定統計量を計算
  4. p値: H₀が真のとき、観測データ以上に極端な結果が出る確率
  5. p値 < α (通常 0.05) なら H₀ を棄却
  
  ■ 2種類のエラー
  第一種過誤 (α): H₀が真なのに棄却 → 偽陽性
  第二種過誤 (β): H₀が偽なのに棄却しない → 偽陰性
  検出力 = 1 - β: 本当の差を見つける力
  
  ■ アナロジー: 裁判
  H₀ = 無罪、H₁ = 有罪
  α = 無実の人を有罪にする確率
  β = 犯人を無罪にする確率

  ⚠ よくある誤解: p値は仮説が正しい確率
  → 正確には: p値は「H₀が真のとき、このデータ以上に極端な値が出る確率」

----------------------------------------------------------------------
  1.4 A/Bテスト — ビジネスで統計を使う代表例
----------------------------------------------------------------------
  先に結論: 「1.4 A/Bテスト — ビジネスで統計を使う代表例」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  A/Bテスト = ランダム化比較試験 (RCT) のWeb版。
  
  設計の要点:
  1. サンプルサイズ計算: 検出したい最小効果量 + 検出力80%以上
  2. ランダム割り当て: ユーザー単位で均等に分割
  3. 十分な期間: 曜日効果・新規性効果を排除
  4. 複数検定補正: テスト項目が増えると偽陽性が増える
     → Bonferroni補正: α/テスト数
  
  よくある落とし穴:
  - ピーキング: テスト中に何度もp値を見て早期終了 → 偽陽性増加
  - Simpson's Paradox: セグメント別では逆の結論になることがある
  - ネットワーク効果: SNSなどでは独立性の仮定が崩れる

  CVR 10% → 10.5% (5%改善) を検出するサンプルサイズ: 57,694/群

  【面接向けに整理すると】
  Q: A/Bテストで統計的有意差が出たが導入すべきか？
  A: 有意差だけでは不十分。実務的有意性 (practical significance) を確認。
     0.01%のCVR改善は有意でも、実装コストに見合わないかもしれない。
     さらにセグメント別分析・長期影響も考慮する。

  2.1 教師あり学習・教師なし学習・強化学習
----------------------------------------------------------------------
  この節の狙い: 「2.1 教師あり学習・教師なし学習・強化学習」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  ┌──────────┬──────────────┬──────────────┬──────────────┐
  │          │ 教師あり     │ 教師なし     │ 強化学習     │
  ├──────────┼──────────────┼──────────────┼──────────────┤
  │ データ   │ (X, y) ペア  │ X のみ       │ 状態→行動→報酬│
  │ 目的     │ y を予測     │ 構造を発見   │ 累積報酬最大 │
  │ 例       │ スパム分類   │ 顧客セグメント│ ゲームAI     │
  │ 評価     │ 精度・AUC    │ シルエット係数│ 累積報酬     │
  └──────────┴──────────────┴──────────────┴──────────────┘
  
  ■ 半教師あり学習: 少量のラベル + 大量の未ラベル
  → Self-Training: 予測が高確信のものを擬似ラベルとして追加
  → 実務で非常に多い (ラベリングは高コスト)
  
  ■ 自己教師あり学習 (Self-Supervised Learning)
  → データ自体からタスクを作る (例: 文の穴埋め = BERT)
  → LLM時代の主流。ラベル不要で大量データを活用。

----------------------------------------------------------------------
  2.2 バイアス-バリアンストレードオフ
----------------------------------------------------------------------
  この節の狙い: 「2.2 バイアス-バリアンストレードオフ」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  予測誤差 = バイアス² + バリアンス + ノイズ
  
  バイアス: モデルが単純すぎて真のパターンを捉えられない (過小適合)
  バリアンス: モデルが複雑すぎてノイズまで学習 (過学習)
  
  ■ アナロジー: 弓矢
  バイアス = 的の中心からの系統的なズレ (照準がずれている)
  バリアンス = 矢のバラつき (手がブレている)
  両方小さいのが理想だが、トレードオフの関係にある。
  
  モデル複雑度  バイアス  バリアンス  訓練誤差  テスト誤差
  低 (線形)     高        低          高         高
  中 (適切)     中        中          中         低 ★
  高 (深いNN)   低        高          低         高

  ⚠ よくある誤解: 複雑なモデルほど良い
  → 正確には: テストデータでの汎化性能が重要。正則化やアンサンブルで制御する

----------------------------------------------------------------------
  2.3 交差検証 — モデル評価の正しい方法
----------------------------------------------------------------------
  この節の狙い: 「2.3 交差検証 — モデル評価の正しい方法」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  なぜ単純なtrain/test分割では不十分か？
  → 分割の仕方による偶然に左右される。
  
  K-Fold Cross Validation:
  1. データをK個に分割
  2. K-1個で訓練、1個でテスト → K回繰り返す
  3. K個のスコアの平均 = 推定汎化性能
  
  注意: 時系列データでは通常のK-Foldは使えない！
  → 未来のデータで訓練して過去を予測してしまう (データリーク)
  → TimeSeriesSplit: 常に過去で訓練、未来でテスト

----------------------------------------------------------------------
  K-Fold デモ (5-Fold)
----------------------------------------------------------------------
  Fold 1: train=[4, 5, 6, 7, 8]...(16件) test=[0, 1, 2, 3](4件)
  Fold 2: train=[0, 1, 2, 3, 8]...(16件) test=[4, 5, 6, 7](4件)
  Fold 3: train=[0, 1, 2, 3, 4]...(16件) test=[8, 9, 10, 11](4件)
  Fold 4: train=[0, 1, 2, 3, 4]...(16件) test=[12, 13, 14, 15](4件)
  Fold 5: train=[0, 1, 2, 3, 4]...(16件) test=[16, 17, 18, 19](4件)

----------------------------------------------------------------------
  2.4 評価指標 — 何を最適化すべきか？
----------------------------------------------------------------------
  まず全体像: この節では「2.4 評価指標 — 何を最適化すべきか？」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  ■ 分類
  Accuracy: 全体の正解率 → 不均衡データでは無意味
  Precision: 陽性予測のうち正解 = TP/(TP+FP) → 偽陽性を減らしたい時
  Recall:    実際の陽性のうち検出 = TP/(TP+FN) → 偽陰性を減らしたい時
  F1:        PrecisionとRecallの調和平均
  AUC-ROC:   閾値非依存の分類能力。ランダム=0.5, 完璧=1.0
  
  ■ 回帰
  MSE: 大きな誤差に敏感 (二乗するから)
  MAE: 外れ値に頑健
  R²:  説明できた分散の割合 (1.0が完璧)
  
  ■ ランキング
  MAP@K, NDCG: 検索・推薦で使用
  
  実務での選び方:
  - 医療診断 → Recall重視 (見逃しが致命的)
  - スパムフィルタ → Precision重視 (正常メールの誤検出は困る)
  - 不均衡データ → AUC-PR (Precision-Recall AUC)

  【面接向けに整理すると】
  Q: Accuracy 98%のモデルは良いモデルか？
  A: データの陽性率が2%なら、全部陰性と予測するだけで98%。
     不均衡データではAUC-ROC、F1、AUC-PRで評価すべき。
     ビジネスKPIとの整合性も重要。

----------------------------------------------------------------------
  3.1 線形回帰 — すべての基礎
----------------------------------------------------------------------
  まず全体像: この節では「3.1 線形回帰 — すべての基礎」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  y = w₁x₁ + w₂x₂ + ... + b
  
  最小二乗法: Σ(yi - ŷi)² を最小化する w, b を求める。
  閉形式解: w = (XᵀX)⁻¹Xᵀy  (正規方程式)
  
  ■ 正則化 — 過学習を防ぐ
  Ridge (L2): 損失 + λΣwi² → 係数を小さくする
  Lasso (L1): 損失 + λΣ|wi| → 不要な特徴量の係数を0にする (特徴選択)
  ElasticNet: L1 + L2 の組合せ
  
  ■ なぜ正則化が必要か？
  特徴量が多いと、訓練データのノイズにも合わせてしまう。
  ペナルティを加えて「シンプルなモデル」に制約する。

  線形回帰結果: y = 1.99x + 0.89 (真の式: y = 2x + 1)

----------------------------------------------------------------------
  3.2 ロジスティック回帰 — 分類の基礎
----------------------------------------------------------------------
  先に結論: 「3.2 ロジスティック回帰 — 分類の基礎」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  線形回帰の出力をシグモイド関数で [0,1] に変換。
  σ(z) = 1 / (1 + e⁻ᶻ)
  
  損失関数: 交差エントロピー (Binary Cross-Entropy)
  L = -Σ[yi log(pi) + (1-yi) log(1-pi)]
  
  ■ なぜMSEではなくCross-Entropyか？
  MSEだと勾配が非常に小さくなり学習が遅い。
  Cross-Entropyは誤った確信に大きなペナルティを与える。
  
  ■ 多クラス化: Softmax回帰
  softmax(zi) = exp(zi) / Σexp(zj) → 全クラスの確率の合計=1
  σ(-3) = 0.0474
  σ(-1) = 0.2689
  σ(+0) = 0.5000
  σ(+1) = 0.7311
  σ(+3) = 0.9526

----------------------------------------------------------------------
  3.3 決定木 — 解釈可能なモデル
----------------------------------------------------------------------
  この節の狙い: 「3.3 決定木 — 解釈可能なモデル」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  「もし〜なら」のルールで分岐する。
  
  分割基準:
  - ジニ不純度: 1 - Σpi² (0 = 純粋、最大 = 混在)
  - エントロピー: -Σpi log₂(pi) (情報理論由来)
  どちらも「分割後に各グループがどれだけ純粋になるか」を測る。
  
  過学習対策:
  - max_depth: 木の深さ制限
  - min_samples_leaf: 葉の最小サンプル数
  - 剪定 (pruning): 事後的に不要な枝を削除
  
  利点: 解釈性が高い、前処理がほぼ不要
  欠点: 単体では精度が低い → アンサンブルで解決

  ジニ不純度: [0.5, 0.5] = 0.50 (最大混在)
  ジニ不純度: [0.9, 0.1] = 0.18 (ほぼ純粋)
  ジニ不純度: [1.0, 0.0] = 0.00 (完全に純粋)

----------------------------------------------------------------------
  3.4 アンサンブル: Random Forest & XGBoost
----------------------------------------------------------------------
  まず全体像: この節では「3.4 アンサンブル: Random Forest & XGBoost」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  ■ Random Forest (バギング)
  - 複数の決定木を独立に訓練、多数決/平均
  - 各木はデータのブートストラップサンプル + 特徴量のランダムサブセット
  - バリアンスを減らす (各木の誤りが相殺)
  
  ■ XGBoost (ブースティング)
  - 木を逐次的に追加。前の木の「間違い」を修正する
  - 勾配降下で残差を最小化 → Gradient Boosting
  - XGBoostの工夫: 正則化項、欠損値の自動処理、並列化
  
  バギング vs ブースティング:
  ┌──────────┬────────────────────┬────────────────────┐
  │          │ バギング (RF)      │ ブースティング(XGB)│
  ├──────────┼────────────────────┼────────────────────┤
  │ 訓練     │ 並列 (独立)        │ 逐次 (依存)        │
  │ 主効果   │ バリアンス削減     │ バイアス削減       │
  │ 過学習   │ しにくい           │ しやすい(要調整)   │
  │ 速度     │ 速い               │ やや遅い           │
  └──────────┴────────────────────┴────────────────────┘

  ⚠ よくある誤解: XGBoostは常にRFより良い
  → 正確には: ハイパーパラメータ調整なしではRFの方が安定することが多い

----------------------------------------------------------------------
  3.5 SVM — マージン最大化
----------------------------------------------------------------------
  まず全体像: この節では「3.5 SVM — マージン最大化」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  サポートベクターマシンは、クラス間の「マージン」を最大化する
  超平面を見つける分類器。
  
  カーネルトリック: 非線形な境界を高次元空間で線形に分離。
  - 線形カーネル: K(x,y) = xᵀy
  - RBFカーネル: K(x,y) = exp(-γ||x-y||²)  ← 最も汎用的
  - 多項式カーネル
  
  特徴: 高次元データ(テキストなど)に強い。
  弱点: 大規模データ(>10万件)で遅い → 今はXGBoostやNNが主流。

----------------------------------------------------------------------
  3.6 k-means クラスタリング
----------------------------------------------------------------------
  まず全体像: この節では「3.6 k-means クラスタリング」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  1. K個の中心をランダム初期化
  2. 各点を最も近い中心に割り当て
  3. 各クラスタの平均を新しい中心に
  4. 収束まで2-3を繰り返す
  
  Kの選び方:
  - エルボー法: クラスタ内分散のグラフで「肘」を探す
  - シルエット係数: クラスタの凝集度と分離度のバランス
  
  限界: 球状クラスタ仮定、Kを事前指定、外れ値に弱い
  → DBSCAN: 密度ベース、Kの指定不要、非球状にも対応

----------------------------------------------------------------------
  3.7 PCA — 次元削減
----------------------------------------------------------------------
  まず全体像: この節では「3.7 PCA — 次元削減」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  主成分分析: データの分散が最大になる方向を見つけ、
  少数の軸でデータを表現する。
  
  手順:
  1. データを標準化
  2. 共分散行列を計算
  3. 固有値分解 → 固有ベクトル = 主成分の方向
  4. 寄与率の高い主成分だけ残す
  
  用途: 可視化 (高次元→2D)、ノイズ除去、特徴量圧縮
  制約: 線形変換のみ → 非線形なら t-SNE, UMAP

  【面接向けに整理すると】
  Q: 100次元の特徴量がある。どう次元削減するか？
  A: まず相関の高い特徴量を確認 (相関行列)。
     PCAで累積寄与率95%まで圧縮。
     ただし解釈性が重要ならLasso/特徴量重要度で選択。
     可視化ならt-SNEかUMAPを使う。

----------------------------------------------------------------------
  4.1 パーセプトロン — 最小のニューロン
----------------------------------------------------------------------
  この節の狙い: 「4.1 パーセプトロン — 最小のニューロン」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  入力 x₁, x₂, ... に重み w₁, w₂, ... を掛けて合計し、
  閾値を超えたら1、そうでなければ0を出力。
  
  z = Σ(wi × xi) + b
  y = 活性化関数(z)
  
  ■ 単層パーセプトロンの限界
  XOR問題を解けない (線形分離不可能)
  → 多層にすることで非線形な決定境界を実現 = MLP

----------------------------------------------------------------------
  4.2 活性化関数 — 非線形性の導入
----------------------------------------------------------------------
  まず全体像: この節では「4.2 活性化関数 — 非線形性の導入」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  なぜ活性化関数が必要か？
  → 線形変換の合成は線形変換。非線形関数がないと深くする意味がない。
  
  ┌────────────┬────────────────┬──────────────────────┐
  │ 関数       │ 定義           │ 特徴                 │
  ├────────────┼────────────────┼──────────────────────┤
  │ Sigmoid    │ 1/(1+e⁻ˣ)     │ 勾配消失問題あり     │
  │ tanh       │ (eˣ-e⁻ˣ)/...  │ 出力が[-1,1]         │
  │ ReLU       │ max(0, x)      │ 現在の標準。Dying問題│
  │ Leaky ReLU │ max(0.01x, x)  │ Dying ReLU対策       │
  │ GELU       │ x·Φ(x)        │ Transformer標準      │
  │ Swish      │ x·σ(x)        │ 自己ゲーティング     │
  └────────────┴────────────────┴──────────────────────┘
  
  Dying ReLU: 負の入力で勾配が常に0 → そのニューロンが「死ぬ」

  x   | ReLU | LeakyReLU | GELU
  -2  | 0.00 |     -0.02 | -0.05
  -1  | 0.00 |     -0.01 | -0.16
  +0  | 0.00 |      0.00 | 0.00
  +1  | 1.00 |      1.00 | 0.84
  +2  | 2.00 |      2.00 | 1.95

----------------------------------------------------------------------
  4.3 勾配降下法 — 学習の仕組み
----------------------------------------------------------------------
  まず全体像: この節では「4.3 勾配降下法 — 学習の仕組み」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  損失関数 L(w) を最小化する重み w を見つける。
  
  w ← w - η × ∂L/∂w   (η = 学習率)
  
  バリエーション:
  - SGD: ミニバッチごとに更新。ノイジーだが高速
  - Momentum: 前回の更新方向を加速 (ボールが坂を転がる)
  - Adam: 学習率を各パラメータごとに適応的に調整
    → 実務のデフォルト。ほぼ全てのDLで使われる
  
  ■ 学習率の重要性
  大きすぎる → 発散 (損失が増加)
  小さすぎる → 収束が遅い or 局所解に嵌る
  → スケジューリング: Warmup + Cosine Decay が主流

----------------------------------------------------------------------
  4.4 バックプロパゲーション — 勾配の効率的な計算
----------------------------------------------------------------------
  まず全体像: この節では「4.4 バックプロパゲーション — 勾配の効率的な計算」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  チェインルール (連鎖律) を使い、出力層から入力層へ
  勾配を「逆方向に」伝播する。
  
  例: f(g(x)) の微分
  df/dx = df/dg × dg/dx
  
  多層NNでは:
  ∂L/∂w₁ = ∂L/∂y × ∂y/∂h₂ × ∂h₂/∂h₁ × ∂h₁/∂w₁
  
  ■ 勾配消失問題
  シグモイドの勾配は最大0.25。層が深いと掛け算で指数的に小さくなる。
  → ReLU (勾配=1 for x>0) で大幅に緩和。
  → 残差接続 (ResNet) でさらに深い学習が可能に。
  
  ■ 勾配爆発
  勾配が指数的に大きくなる → Gradient Clipping で制御

  ⚠ よくある誤解: NNはブラックボックスなので理論は不要
  → 正確には: 勾配の流れを理解しないと、学習が上手くいかない時のデバッグが不可能

  【面接向けに整理すると】
  Q: BatchNorm はなぜ効くのか？
  A: 各層の入力の分布を正規化し、内部共変量シフトを軽減。
     学習率を大きくできる、初期化への依存が減る、
     軽い正則化効果もある。ただし「なぜ効くか」の完全な
     理論的説明は今も研究中。

  5.1 CNN — 画像認識の革命
----------------------------------------------------------------------
  先に結論: 「5.1 CNN — 画像認識の革命」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  Convolutional Neural Network: 画像の局所パターンを階層的に学習。
  
  ■ 核心アイデア: 畳み込み
  小さなフィルタ (3×3 など) をスライドさせ、局所特徴を抽出。
  パラメータ共有 → 全結合に比べパラメータ数が劇的に少ない。
  
  構成要素:
  Conv層:    フィルタで特徴マップを生成
  Pooling層: 空間サイズを縮小 (Max Pooling が一般的)
  FC層:      最終的な分類
  
  発展:
  LeNet → AlexNet → VGG → ResNet → EfficientNet → ViT
  
  ■ ResNetの革新: スキップ接続
  y = F(x) + x  ← 入力 x をそのまま加算
  → 100層以上の学習が可能に (勾配が直接流れる)

----------------------------------------------------------------------
  CNN出力サイズ計算
----------------------------------------------------------------------
  Conv3x3,s1,p1        → 224×224
  MaxPool2x2,s2        → 112×112
  Conv3x3,s1,p1        → 112×112
  MaxPool2x2,s2        → 56×56

----------------------------------------------------------------------
  5.2 RNN / LSTM — 系列データ
----------------------------------------------------------------------
  この節の狙い: 「5.2 RNN / LSTM — 系列データ」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  Recurrent Neural Network: 時系列データを扱う。
  隠れ状態 h_t = f(W_h × h_{t-1} + W_x × x_t)
  
  問題: 長い系列で勾配消失 → 過去の情報を忘れる。
  
  ■ LSTM (Long Short-Term Memory)
  3つのゲートで情報の流れを制御:
  - 忘却ゲート: 過去の情報をどれだけ忘れるか
  - 入力ゲート: 新しい情報をどれだけ記憶するか
  - 出力ゲート: 何を出力するか
  
  ■ GRU: LSTMの簡略版 (ゲート2つ)。性能はほぼ同等。
  
  現状: ほとんどのタスクでTransformerに置き換えられた。
  ただしエッジデバイスなど計算制約がある場面では今も有用。

----------------------------------------------------------------------
  5.3 Attention メカニズム
----------------------------------------------------------------------
  先に結論: 「5.3 Attention メカニズム」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  ■ なぜ必要か？
  RNNは系列全体を固定長ベクトルに圧縮 → 情報のボトルネック。
  Attentionは「今どこを見るべきか」を動的に決定する。
  
  Scaled Dot-Product Attention:
  Attention(Q, K, V) = softmax(QKᵀ / √d_k) × V
  
  Q (Query): 「何を探しているか」
  K (Key):   「何を持っているか」
  V (Value): 「実際の内容」
  
  ■ アナロジー: 図書館
  Q = 「機械学習の入門書が欲しい」
  K = 各本のラベル・タグ
  V = 本の中身
  QとKのマッチ度で、どの本(V)を重点的に読むか決める。

----------------------------------------------------------------------
  Attention重みの計算デモ
----------------------------------------------------------------------
  Key0: score=0.707, attention_weight=0.434
  Key1: score=0.000, attention_weight=0.214
  Key2: score=0.495, attention_weight=0.351

----------------------------------------------------------------------
  5.4 Transformer — 現代AIの基盤
----------------------------------------------------------------------
  この節の狙い: 「5.4 Transformer — 現代AIの基盤」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  "Attention Is All You Need" (2017) で提案。
  RNNを完全に排除し、Self-Attentionだけで系列を処理。
  
  ■ 構成
  Encoder: 入力系列を文脈付き表現に変換
  Decoder: 出力系列を自己回帰的に生成
  
  ■ 核心的な利点
  1. 並列化: RNNと違い全位置を同時に処理 → GPU効率が高い
  2. 長距離依存: 任意の位置間の関係を直接捉える
  3. スケーラビリティ: パラメータを増やすほど性能向上
  
  ■ Multi-Head Attention
  異なる「観点」で複数のAttentionを並列実行。
  1つのヘッドは構文、別のヘッドは意味、等を学習。
  
  ■ 位置エンコーディング
  Attentionは順序を知らない → 位置情報を明示的に追加
  正弦波 or 学習可能な位置埋め込み

  ⚠ よくある誤解: TransformerはNLPだけの技術
  → 正確には: Vision (ViT), 音声 (Whisper), タンパク質 (AlphaFold) 等、全領域で使用

  【面接向けに整理すると】
  Q: Transformerが RNN/LSTMに勝る理由は？
  A: (1) 全位置を並列処理 → 訓練が高速
     (2) Self-Attentionで長距離の依存関係を直接モデル化
     (3) スケーリング則: モデル・データを大きくすると予測可能に性能向上
     ただしAttentionの計算量はO(n²)で、長い系列にはMamba等の代替が研究中。

----------------------------------------------------------------------
  6.1 GPTアーキテクチャ
----------------------------------------------------------------------
  先に結論: 「6.1 GPTアーキテクチャ」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  GPT = Generative Pre-trained Transformer
  Transformer の Decoder のみを使用。
  
  訓練: 次のトークンを予測 (Causal Language Model)
  P(x_t | x_1, ..., x_{t-1}) を最大化
  
  ■ スケーリング則 (Kaplan et al., 2020)
  性能はモデルサイズ・データ量・計算量のべき乗則に従う。
  → 大きくすれば予測可能に改善 (ただしコストも指数的に増加)
  
  GPT-4o, Claude, Gemini 等の最新モデルは数千億〜兆パラメータ。

----------------------------------------------------------------------
  6.2 トークナイゼーション
----------------------------------------------------------------------
  まず全体像: この節では「6.2 トークナイゼーション」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  テキストをモデルが処理できる最小単位に分割する。
  
  BPE (Byte Pair Encoding):
  1. 全文字をトークンとして開始
  2. 最も頻繁に隣接するペアをマージ
  3. 語彙サイズに達するまで繰り返す
  
  例: "lower" → ["low", "er"] or ["l", "ow", "er"]
  
  ■ なぜ重要か？
  - トークン数 = コスト (API課金はトークン単位)
  - 日本語は英語の約1.5-2倍のトークン数になりがち
  - 未知語の扱い: サブワードなので完全な未知語は少ない

----------------------------------------------------------------------
  BPEの直感的デモ
----------------------------------------------------------------------
  入力: aaabdaaabac
  Step 1: ('a', 'a') → 'aa' → ['aa', 'a', 'b', 'd', 'aa', 'a', 'b', 'a', 'c']
  Step 2: ('aa', 'a') → 'aaa' → ['aaa', 'b', 'd', 'aaa', 'b', 'a', 'c']
  Step 3: ('aaa', 'b') → 'aaab' → ['aaab', 'd', 'aaab', 'a', 'c']

----------------------------------------------------------------------
  6.3 ファインチューニング手法
----------------------------------------------------------------------
  まず全体像: この節では「6.3 ファインチューニング手法」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  ■ Full Fine-Tuning
  全パラメータを更新。最高精度だがコストが高い。
  
  ■ LoRA (Low-Rank Adaptation)
  重み行列 W を W + BA で近似 (B, A は低ランク行列)。
  元のWは凍結し、B, A だけ学習 → パラメータ数を99%以上削減。
  
  例: W が 4096×4096 の場合
  Full: 16M パラメータ
  LoRA (rank=8): 4096×8 + 8×4096 = 65K パラメータ
  
  ■ QLoRA: 量子化 + LoRA
  ベースモデルを4bit量子化してメモリ削減 + LoRAで学習。
  1枚のGPUで70Bモデルのファインチューニングが可能に。
  
  ■ 使い分け
  データ < 100件: Few-shot prompting
  データ 100-10K: LoRA / QLoRA
  データ > 10K + 予算あり: Full Fine-Tuning

----------------------------------------------------------------------
  6.4 RLHF & DPO — 人間の好みに合わせる
----------------------------------------------------------------------
  この節の狙い: 「6.4 RLHF & DPO — 人間の好みに合わせる」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  ■ RLHF (Reinforcement Learning from Human Feedback)
  1. SFT: 教師ありファインチューニング (良い回答例で学習)
  2. 報酬モデル: 人間が回答ペアに優劣をつけて訓練
  3. PPO: 報酬モデルのスコアを最大化するようRL
  
  ■ DPO (Direct Preference Optimization)
  報酬モデルなしで、人間の好みデータから直接最適化。
  RLHFより安定・シンプル。最近の主流。
  
  損失: L = -log σ(β(log π(yw|x)/πref(yw|x) - log π(yl|x)/πref(yl|x)))
  yw = 好まれた回答, yl = 好まれなかった回答
  
  ■ なぜ必要か？
  事前学習だけでは有害・不正確な応答をする。
  人間の価値観・期待に「整列 (alignment)」させる必要がある。

----------------------------------------------------------------------
  6.5 プロンプトエンジニアリング
----------------------------------------------------------------------
  まず全体像: この節では「6.5 プロンプトエンジニアリング」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  ■ 基本テクニック
  - Zero-shot: 例なしで指示
  - Few-shot: 数例を提示してパターンを示す
  - Chain-of-Thought (CoT): 「ステップごとに考えて」→ 推論精度向上
  - Self-Consistency: 複数回推論して多数決
  
  ■ 構造化プロンプト
  - System / User / Assistant のロール分離
  - XML/JSON で出力形式を指定
  - 制約条件を明確に列挙
  
  ■ 高度なテクニック
  - ReAct: 思考→行動→観察のループ
  - Tree of Thoughts: 推論の分岐を探索
  - Reflexion: 自己反省による改善

  ⚠ よくある誤解: プロンプトが長いほど良い結果が出る
  → 正確には: 冗長なプロンプトはノイズになる。構造化と簡潔さが重要

  【面接向けに整理すると】
  Q: LoRAとFull Fine-Tuningの使い分けは？
  A: LoRAは低コスト・低リスクで、ほとんどのユースケースで十分。
     ドメイン知識が大幅に異なる場合 (例: 医療特化) や
     タスク性能を最後の数%まで追求する場合にFull FTを検討。
     まずLoRAで始めて、不十分ならFull FTに移行するのが実務的。

  7.1 RAG — 検索拡張生成
----------------------------------------------------------------------
  先に結論: 「7.1 RAG — 検索拡張生成」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  Retrieval-Augmented Generation:
  LLMの知識の限界を外部検索で補完する。
  
  基本フロー:
  クエリ → Embedding → ベクトル検索 → 関連文書取得 → LLMに渡して生成
  
  ■ なぜRAGか？
  1. 最新情報への対応 (LLMの学習データは古い)
  2. ハルシネーション軽減 (根拠に基づく回答)
  3. コスト: ファインチューニングより安い
  4. 出典の提示が可能

----------------------------------------------------------------------
  7.2 ベクトル検索
----------------------------------------------------------------------
  先に結論: 「7.2 ベクトル検索」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  テキストを高次元ベクトル (embedding) に変換し、
  コサイン類似度や内積で近いものを検索。
  
  cos(A, B) = (A·B) / (||A|| × ||B||)
  
  ■ インデックスアルゴリズム
  - Flat: 全ベクトルと比較 → 正確だが遅い O(n)
  - IVF: 空間を分割して近傍だけ検索 → 近似だが速い
  - HNSW: グラフベースの近似最近傍 → 精度と速度のバランスが良い
  
  ■ ベクトルDB
  Pinecone, Weaviate, Qdrant, pgvector, Chroma 等。
  メタデータフィルタリングも重要 (日付・カテゴリで絞り込み)。

  cos(猫, 犬) = 0.984  ← 意味的に近い
  cos(猫, 車) = 0.110  ← 意味的に遠い

----------------------------------------------------------------------
  7.3 チャンキング戦略
----------------------------------------------------------------------
  この節の狙い: 「7.3 チャンキング戦略」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  文書を検索単位に分割する。RAGの品質を大きく左右する。
  
  ┌──────────────┬────────────────────┬───────────────────┐
  │ 手法         │ 特徴               │ 適用場面          │
  ├──────────────┼────────────────────┼───────────────────┤
  │ 固定長       │ 文字数で分割       │ シンプルだが文脈断│
  │ セマンティック│ 意味の切れ目で分割 │ 高品質だがコスト高│
  │ 再帰的       │ 段落→文→文字で階層 │ LangChainのデフォ │
  │ 親子構造     │ 大チャンク→小チャン│ コンテキスト保持  │
  └──────────────┴────────────────────┴───────────────────┘
  
  ■ Advanced RAG テクニック
  - Hybrid Search: ベクトル検索 + BM25 (キーワード) の組合せ
  - Reranking: Cross-Encoderで再スコアリング → 精度向上
  - Query Expansion: クエリをLLMで拡張して検索
  - HyDE: 仮説回答を生成してから検索 (回答の分布に寄せる)

----------------------------------------------------------------------
  7.4 AIエージェント — ReAct & Tool Use
----------------------------------------------------------------------
  まず全体像: この節では「7.4 AIエージェント — ReAct & Tool Use」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  エージェント = LLM + 推論 + ツール呼び出し + メモリ
  
  ■ ReAct パターン
  Thought: 何をすべきか考える
  Action:  ツールを選択して実行
  Observation: 結果を観察
  → 目標達成まで繰り返す
  
  ■ Tool Use (Function Calling)
  LLMが「どのツールを、どの引数で呼ぶか」をJSON等で出力。
  ツール: Web検索、DB問合せ、API呼出、コード実行 等
  
  ■ Planning パターン
  - Plan-and-Execute: まず計画を立て、ステップごとに実行
  - Reflexion: 実行結果を振り返り計画を修正

----------------------------------------------------------------------
  7.5 Multi-Agent システム
----------------------------------------------------------------------
  この節の狙い: 「7.5 Multi-Agent システム」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  複数のエージェントが協調して複雑なタスクを解決。
  
  パターン:
  1. Supervisor: 統括エージェントが子エージェントに指示
  2. Debate: 複数エージェントが議論して結論を出す
  3. Pipeline: 直列に処理を渡す (リサーチ→執筆→校正)
  
  ■ フレームワーク
  - LangGraph: 状態グラフでエージェントフローを定義
  - CrewAI: 役割ベースのマルチエージェント
  - AutoGen: Microsoftの対話型マルチエージェント
  
  ■ 課題
  - エラーの伝播: 1つのエージェントの誤りが連鎖する
  - コスト: API呼出が増え、コストが予測しづらい
  - 評価の難しさ: 中間ステップの品質をどう測るか

  ⚠ よくある誤解: エージェントは何でも自律的にやらせるべき
  → 正確には: Human-in-the-Loopが重要。重要な判断にはユーザー確認を入れる

  【面接向けに整理すると】
  Q: RAGとファインチューニング、どちらを選ぶか？
  A: RAG = 知識の追加 (最新情報、社内文書、出典重要)
     FT = 振る舞いの変更 (文体、フォーマット、ドメイン適応)
     多くの場合RAGを先に試す。両方の組合せが最強。
     RAGはデータ更新が容易、FTは再学習が必要。

  8.1 MLOpsとは何か
----------------------------------------------------------------------
  先に結論: 「8.1 MLOpsとは何か」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  ML + DevOps = MLOps
  モデルを「実験」から「本番運用」に持っていくための実践体系。
  
  ■ なぜ必要か？

  Level 1: MLパイプライン自動化 (自動再学習)
  Level 2: CI/CD for ML (テスト・バリデーション自動化)

  8.2 実験管理 — MLflow

  先に結論: 「8.2 実験管理 — MLflow」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  MLflow は実験管理のデファクトスタンダード。
  
  4つのコンポーネント:
  1. Tracking: パラメータ・メトリクス・アーティファクトの記録
  2. Projects: 再現可能なML実行環境の定義
  3. Models: モデルのパッケージング・デプロイ
  4. Registry: モデルのバージョン管理・ステージ遷移

  mlflow.log_param("learning_rate", 0.01)

----------------------------------------------------------------------
  8.3 Feature Store
----------------------------------------------------------------------
  この節の狙い: 「8.3 Feature Store」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  特徴量の計算・保存・再利用を一元管理する仕組み。
  
  ■ なぜ必要か？
  - 訓練と推論で特徴量の計算ロジックが違う → バグの温床
  - チーム間で同じ特徴量を再計算 → 無駄
  - ポイントインタイム正確性: 過去時点の正確な特徴量が必要
  
  主要ツール: Feast (OSS), Tecton, SageMaker Feature Store
  
  オフラインストア: 訓練用 (バッチ、S3/BigQuery等)
  オンラインストア: 推論用 (低レイテンシ、Redis/DynamoDB等)

----------------------------------------------------------------------
  8.4 モデル監視 — ドリフト検出
----------------------------------------------------------------------
  先に結論: 「8.4 モデル監視 — ドリフト検出」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  本番デプロイ後、モデルの性能は時間とともに劣化する。
  
  ■ ドリフトの種類
  1. データドリフト: 入力データの分布が変化
     例: コロナで購買パターンが激変
  2. コンセプトドリフト: X→Yの関係性が変化
     例: 「リモートワーク」の求人カテゴリが変化
  3. 予測ドリフト: モデル出力の分布が変化
  
  ■ 検出方法
  - KSテスト: 2つの分布の差を検定
  - PSI (Population Stability Index): 分布の安定性指標
  - パフォーマンス監視: 精度メトリクスの推移
  
  ■ 対策
  - 自動再学習パイプライン (定期 or トリガーベース)
  - Shadow deployment: 新モデルを本番トラフィックで評価
  - A/Bテスト: 新旧モデルを比較

----------------------------------------------------------------------
  PSI計算デモ
----------------------------------------------------------------------
  PSI (安定データ): 0.0665 → OK
  PSI (ドリフトデータ): 1.2771 → 要注意!
  判定基準: <0.1=安定, 0.1-0.25=やや変化, >0.25=大きな変化

  【面接向けに整理すると】
  Q: モデルの精度が徐々に低下している。原因と対策は？
  A: 1. データドリフトを確認 (PSI, KSテスト)
     2. 入力データの品質チェック (欠損、異常値)
     3. コンセプトドリフトか確認 (ラベルの意味が変わってないか)
     4. 対策: 再学習、特徴量の追加・見直し、ウィンドウの調整

  Chapter 9: コンピュータビジョン & 生成AI

----------------------------------------------------------------------
  9.1 物体検出 — YOLO
----------------------------------------------------------------------
  先に結論: 「9.1 物体検出 — YOLO」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  画像内の物体の位置 (バウンディングボックス) とクラスを同時に予測。
  
  ■ 2段階 vs 1段階
  2段階 (Faster R-CNN): 領域提案→分類 → 高精度だが遅い
  1段階 (YOLO, SSD):    画像全体を一度に処理 → 高速
  
  ■ YOLO (You Only Look Once)
  画像をグリッドに分割し、各セルが物体を予測。
  v1 → v8 → v11 (YOLO11) と進化。
  
  ■ アンカーボックス
  事前定義した複数のボックス形状をテンプレートとして使用。
  モデルはアンカーからのオフセットを予測。
  
  ■ NMS (Non-Maximum Suppression)
  重複する予測ボックスを除去。IoU(重なり率)で判定。
  
  IoU = 重なり面積 / 合計面積

  IoU([0,0,10,10], [5,5,15,15]) = 0.14
  IoU(同一ボックス) = 1.00

----------------------------------------------------------------------
  9.2 セグメンテーション
----------------------------------------------------------------------
  先に結論: 「9.2 セグメンテーション」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  ■ セマンティックセグメンテーション
  各ピクセルにクラスを割り当て (空、道路、建物...)
  U-Net (医療画像), DeepLab, SegFormer
  
  ■ インスタンスセグメンテーション
  個々の物体を区別 (犬A, 犬B を別インスタンスとして認識)
  Mask R-CNN, SAM (Segment Anything Model)
  
  ■ SAM (Meta, 2023)
  あらゆる画像に対してゼロショットでセグメンテーション。
  プロンプト (点、ボックス、テキスト) で対象を指定。
  → 画像版の「Foundation Model」。

----------------------------------------------------------------------
  9.3 Diffusion Models — 画像生成の革命
----------------------------------------------------------------------
  この節の狙い: 「9.3 Diffusion Models — 画像生成の革命」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  ■ 核心アイデア
  画像にノイズを段階的に加え (Forward Process)、
  そのノイズを除去する方法を学習 (Reverse Process)。
  
  Forward: x_0 → x_1 → ... → x_T (ガウスノイズ)
  Reverse: x_T → x_{T-1} → ... → x_0 (生成画像)
  
  ■ アナロジー: 彫刻
  大理石のブロック (ノイズ) から不要な部分を削っていく。
  各ステップで少しずつノイズを除去 → 画像が浮かび上がる。
  
  ■ 主要モデル
  - DALL-E 3: テキスト→画像 (OpenAI)
  - Stable Diffusion: オープンソース、Latent空間で動作
  - Midjourney: 高品質アート生成
  
  ■ Latent Diffusion (Stable Diffusion)
  画像空間 (512×512×3) ではなく潜在空間 (64×64×4) で
  Diffusionを行う → 計算コストを大幅に削減。
  VAE Encoder で画像→潜在表現、Decoder で復元。
  
  ■ ControlNet: 構造を制御
  エッジ画像、深度マップ、ポーズ等で生成を制御。
  → 「構図はこれ、スタイルはあれ」が可能に。

  ⚠ よくある誤解: Diffusionモデルはランダムなノイズから画像を作る
  → 正確には: 条件付き生成。テキストプロンプトが「どのノイズを除去するか」を導く

  【面接向けに整理すると】
  Q: GANとDiffusion Modelの違いは？
  A: GAN: Generator vs Discriminator の対戦。訓練が不安定 (モード崩壊)。
     Diffusion: ノイズ除去を学習。訓練が安定、多様性が高い。
     現在の画像生成はDiffusionが主流。ただしGANは生成速度が速い。
     リアルタイム用途ではGANやDistillation版Diffusionも使われる。

  Chapter 10: 評価 & ガードレール — LLMアプリの品質と安全性

----------------------------------------------------------------------
  10.1 RAGAS — RAG評価フレームワーク
----------------------------------------------------------------------
  先に結論: 「10.1 RAGAS — RAG評価フレームワーク」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  RAGシステムの品質を自動評価するメトリクス群。
  
  ■ Faithfulness (忠実性)
  回答がコンテキスト (検索結果) に基づいているか？
  1. 回答をクレーム (主張) に分解
  2. 各クレームがコンテキストから裏付けられるか判定
  スコア = 裏付けされたクレーム数 / 全クレーム数
  
  ■ Answer Relevancy (回答関連性)
  回答がクエリに適切に答えているか？
  回答からクエリを逆生成し、元クエリとの類似度で評価。
  
  ■ Context Precision (文脈精度)
  検索されたコンテキストが関連性の高い順に並んでいるか？
  
  ■ Context Recall (文脈再現率)
  正解に必要な情報がコンテキストに含まれているか？

----------------------------------------------------------------------
  Faithfulness計算デモ
----------------------------------------------------------------------
  ✓ 裏付けあり: 東京は日本の首都
  ✓ 裏付けあり: 人口は約1400万人
  ✗ 裏付けなし: 面積は世界最大
  Faithfulness = 0.67

----------------------------------------------------------------------
  10.2 DeepEval — LLM評価
----------------------------------------------------------------------
  まず全体像: この節では「10.2 DeepEval — LLM評価」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  LLMアプリケーションの包括的な評価フレームワーク。
  
  ■ 主要メトリクス
  - Hallucination: 事実と異なる内容の生成度合い
  - Toxicity: 有害コンテンツの含有度
  - Bias: 偏見・差別的な内容の度合い
  - Summarization: 要約の品質 (重要情報のカバー率)
  
  ■ LLM-as-Judge
  LLM自体を評価者として使う手法。
  課題:
  - Position Bias: 先に提示された回答を高く評価しがち
  - Verbosity Bias: 長い回答を高く評価しがち
  - Self-Enhancement Bias: 同じモデルの出力を高く評価
  対策: 順序入れ替え、複数回評価、異なるモデルで評価

----------------------------------------------------------------------
  10.3 NeMo Guardrails — 対話の安全制御
----------------------------------------------------------------------
  まず全体像: この節では「10.3 NeMo Guardrails — 対話の安全制御」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  NVIDIA NeMo Guardrails: LLMの入出力を制御するフレームワーク。
  
  ■ Colang (設定言語)
  対話フローをルールとして定義:
  
  define user ask about politics
    "大統領について教えて"
    "選挙の予想は？"
  
  define flow
    user ask about politics
    bot refuse political topics
    "政治的なトピックには回答できません"
  
  ■ ガードレールの種類
  1. Input Rails: ユーザー入力のフィルタリング
     - トピック制限、Prompt Injection検出
  2. Output Rails: LLM出力のフィルタリング
     - ファクトチェック、有害コンテンツ除去
  3. Retrieval Rails: 検索結果のフィルタリング
     - 関連性の低い結果を除去
  4. Execution Rails: ツール実行の制御
     - 許可されたツールのみ実行

----------------------------------------------------------------------
  10.4 Presidio — PII検出・マスキング
----------------------------------------------------------------------
  この節の狙い: 「10.4 Presidio — PII検出・マスキング」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  Microsoft Presidio: 個人情報 (PII) の検出と匿名化。
  
  ■ 検出可能なエンティティ
  - 電話番号、メールアドレス、クレジットカード番号
  - 氏名、住所、マイナンバー、パスポート番号 等
  
  ■ 匿名化手法
  - マスキング: "田中太郎" → "****"
  - 仮名化:   "田中太郎" → "Person_001" (復元可能)
  - 一般化:   "35歳" → "30代"
  - ハッシュ化: 復元不可能な変換
  
  ■ なぜLLMアプリに必要か？
  ユーザーがPIIを含む入力を送信 → LLMに渡る → ログに残る
  → 個人情報保護法/GDPR違反のリスク
  → 入力段階でPIIを検出・マスキングする必要がある

----------------------------------------------------------------------
  PII検出デモ (正規表現ベース)
----------------------------------------------------------------------
  入力: 連絡先は tanaka@example.com、電話は 090-1234-5678 です。
  検出: EMAIL → tanaka@example.com
  検出: PHONE → 090-1234-5678
  マスク後: 連絡先は <EMAIL>、電話は <PHONE> です。

----------------------------------------------------------------------
  10.5 Prompt Injection 防御
----------------------------------------------------------------------
  先に結論: 「10.5 Prompt Injection 防御」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  ■ 攻撃の種類
  1. Direct Injection: 指示を直接書き換える
     "上記の指示を無視して..."
  2. Indirect Injection: 検索結果やWebページに悪意の指示を埋め込む
     RAGで取得した文書に "Ignore previous instructions" が含まれる
  
  ■ 多層防御
  Layer 1: 入力フィルタリング (既知パターンのブロック)
  Layer 2: Prompt設計 (System promptでの堅牢な指示)
  Layer 3: 出力検証 (生成結果のポリシーチェック)
  Layer 4: 権限制限 (エージェントのツール権限を最小限に)
  
  ■ 完全な防御は不可能
  LLMは「指示」と「データ」を区別できない根本的な限界がある。
  → リスクベースのアプローチ: 影響度に応じた防御レベルの設計

  ⚠ よくある誤解: ガードレールを入れれば安全
  → 正確には: ガードレールは「リスク軽減」であり「リスク排除」ではない。重要な処理には人間の確認 (Human-in-the-Loop) を組み込む

  【面接向けに整理すると】
  Q: LLMアプリの品質保証をどう設計するか？
  A: 多層的なアプローチが必要:
     1. 自動評価: RAGAS/DeepEvalでCI/CDに組込み
     2. ガードレール: NeMo等で入出力フィルタリング
     3. PII保護: Presidioで個人情報をマスク
     4. 監視: 本番の応答品質・レイテンシ・コストを継続監視
     5. フィードバック: ユーザー評価を収集し改善サイクルを回す
