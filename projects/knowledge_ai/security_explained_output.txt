======================================================================
  Chapter 1: 暗号学基礎 — 対称鍵・非対称鍵・鍵交換
======================================================================

----------------------------------------------------------------------
  なぜ暗号が必要か
----------------------------------------------------------------------
  まず前提:
  ネットワーク上のデータは、平文のままだと第三者に読み取られうる。
  暗号は「盗聴できる環境でも安全に通信する」ための基盤技術。
  
  暗号が担う目的は1つではない。代表はCIAの3軸:
  - Confidentiality（機密性）: 読まれては困る情報を守る
  - Integrity（完全性）: 書き換えられていないことを保証する
  - Availability（可用性）: 必要時に使える状態を維持する
  
  実務で特に重要なのは、機密性だけを見ないこと。
  たとえば「暗号化はしているが改ざん検知がない」設計は危険。
  現代のプロトコルは、暗号化と認証をセットで扱うのが基本になる。
  
  たとえ話:
  暗号化は封筒、認証は封印、鍵管理は鍵束の保管ルール。
  どれか1つ欠けても安全性は成立しない。

----------------------------------------------------------------------
  対称鍵暗号 — AES
----------------------------------------------------------------------
  先に結論: 「対称鍵暗号 — AES」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  「同じ鍵」で暗号化と復号を行う方式。家の鍵のイメージ。
  
  AES（Advanced Encryption Standard）:
  - ブロック暗号。128ビット単位でデータを処理
  - 鍵長: 128/192/256ビット（256が最も安全）
  - 速い。ハードウェアアクセラレーション（AES-NI）対応
  
  動作モード（なぜブロック暗号にモードが必要？）:
  - ECB: 各ブロックを独立暗号化。同じ平文→同じ暗号文になり危険
  - CBC: 前のブロックの暗号文をXOR。パディングオラクル攻撃に脆弱
  - GCM: カウンターモード＋認証タグ。改ざん検知もできる。現代の標準
  - GCMが最も推奨される理由: 暗号化と認証を同時に行う（AEAD）

  ⚠ よくある誤解: AES-256は絶対に破れない
  → 正確には: アルゴリズム自体は安全だが、実装の脆弱性（サイドチャネル攻撃、不適切なIV管理）で破られることがある。暗号は実装が9割

----------------------------------------------------------------------
  非対称鍵暗号 — RSA と ECC
----------------------------------------------------------------------
  先に結論: 「非対称鍵暗号 — RSA と ECC」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  「公開鍵」と「秘密鍵」のペアを使う方式。
  アナロジー: 南京錠（公開鍵）を配り、鍵（秘密鍵）は自分だけが持つ。
  
  RSA:
  - 巨大な素数2つの積を因数分解するのが困難という数学的性質を利用
  - 鍵長2048ビット以上が必須（1024は既に危険）
  - 欠点: 鍵が大きい、計算が遅い
  
  ECC（楕円曲線暗号）:
  - 楕円曲線上の離散対数問題を利用
  - RSA 2048ビット ≈ ECC 256ビット（同等の安全性で鍵が小さい）
  - モバイル・IoTで特に有利（計算資源が少ない）
  - 代表的な曲線: P-256, Curve25519
  
  使い分け:
  - データの暗号化 → 対称鍵（速い）
  - 鍵の交換・署名 → 非対称鍵（鍵配送問題を解決）
  - 実際のTLS通信: 非対称鍵で対称鍵を安全に共有→以降は対称鍵で通信

----------------------------------------------------------------------
  Diffie-Hellman 鍵交換
----------------------------------------------------------------------
  この節の狙い: 「Diffie-Hellman 鍵交換」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  目的: 盗聴されている回線上で、安全に「共通の秘密」を作る。
  
  アナロジー（色の混合）:
  1. AliceとBobが「黄色」を公開で共有（公開パラメータ）
  2. Aliceは秘密の色「赤」を混ぜる → オレンジを送信
  3. Bobは秘密の色「青」を混ぜる → 緑を送信
  4. Aliceはオレンジに青を混ぜる → 茶色
  5. Bobは緑に赤を混ぜる → 同じ茶色
  → 盗聴者はオレンジと緑は見えるが、茶色は作れない
  
  ECDHE（Elliptic Curve Diffie-Hellman Ephemeral）:
  - 楕円曲線版DH + 一時鍵（Ephemeral）
  - 一時鍵を毎回生成するため「前方秘匿性」を実現
  - TLS 1.3の標準鍵交換方式

  ⚠ よくある誤解: DHは暗号化のアルゴリズム
  → 正確には: DHは鍵交換のプロトコル。データの暗号化は行わない。DHで共有した秘密をAES等の鍵として使う

----------------------------------------------------------------------
  前方秘匿性（Forward Secrecy）
----------------------------------------------------------------------
  先に結論: 「前方秘匿性（Forward Secrecy）」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  秘密鍵が将来漏洩しても、過去の通信は復号できない性質。
  
  なぜ重要か:
  - 攻撃者が暗号文を全て保存しておき、将来鍵を入手して復号する攻撃
  - 一時鍵（Ephemeral Key）を毎セッション生成し、終了後に破棄
  - 長期鍵の漏洩が過去の通信に影響しない
  
  量子コンピュータの脅威:
  - Shorのアルゴリズム: RSA・ECCを破る可能性
  - 対策: 格子暗号等のポスト量子暗号（PQC）の標準化が進行中
  - NIST PQC標準: CRYSTALS-Kyber（鍵交換）, CRYSTALS-Dilithium（署名）

  【面接向けに整理すると】
  Q: 対称鍵と非対称鍵の違いと使い分けは？
  A: 対称鍵は同一鍵で暗号化・復号（高速だが鍵配送が課題）、
     非対称鍵は公開鍵・秘密鍵ペア（低速だが鍵配送問題を解決）。
     実務ではハイブリッド: 非対称鍵で対称鍵を交換し、以降は対称鍵で通信。
     TLS 1.3ではECDHEで鍵交換、AES-256-GCMでデータ暗号化が典型。

======================================================================
  Chapter 2: ハッシュ & PKI — 完全性と信頼の仕組み
======================================================================

----------------------------------------------------------------------
  ハッシュ関数とは
----------------------------------------------------------------------
  この節の狙い: 「ハッシュ関数とは」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  任意長の入力 → 固定長の出力（ダイジェスト）への一方向変換。
  
  性質（暗号学的ハッシュの3要件）:
  1. 衝突耐性: 同じハッシュ値を持つ2つの入力を見つけるのが困難
  2. 原像耐性: ハッシュ値から元の入力を復元するのが困難
  3. 第二原像耐性: ある入力と同じハッシュ値の別入力を見つけるのが困難
  
  SHA-256:
  - SHA-2ファミリー。256ビット出力
  - ファイル改ざん検知、デジタル署名、ブロックチェーンで使用
  - SHA-1は衝突が発見済み → 使ってはいけない

----------------------------------------------------------------------
  パスワードハッシュ — bcrypt / Argon2
----------------------------------------------------------------------
  まず全体像: この節では「パスワードハッシュ — bcrypt / Argon2」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  なぜSHA-256でパスワードを保存してはいけないのか？
  → SHA-256は「速い」。GPUで毎秒数十億回計算でき、総当たりが容易。
  
  パスワードハッシュの要件:
  - 意図的に遅くする（計算コストを調整可能）
  - ソルト付き（同じパスワードでも異なるハッシュ値）
  - メモリハード（GPU/ASICでの並列攻撃を困難にする）
  
  bcrypt:
  - Blowfish暗号ベース。コストファクターで計算量を調整
  - ソルト自動付与。広く使われている
  - 入力72バイト制限がある点に注意
  
  Argon2:
  - 2015年Password Hashing Competition優勝
  - メモリ使用量・並列度・反復回数を個別に調整可能
  - Argon2id: サイドチャネル攻撃とGPU攻撃の両方に耐性
  - 新規開発ではArgon2idを推奨

  ⚠ よくある誤解: ハッシュは暗号化の一種
  → 正確には: ハッシュは一方向関数であり復号できない。暗号化は復号を前提とする。パスワードは『暗号化して保存』ではなく『ハッシュして保存』が正しい

----------------------------------------------------------------------
  PKI — 公開鍵基盤
----------------------------------------------------------------------
  まず全体像: この節では「PKI — 公開鍵基盤」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  問題: 公開鍵が本当にその人のものか、どう確認する？
  → 信頼できる第三者（CA: 認証局）が「この公開鍵はこの人のもの」と証明。
  
  証明書チェーン:
  ルートCA → 中間CA → サーバー証明書
  
  - ルートCA: OSやブラウザに事前インストール（トラストストア）
  - 中間CA: ルートCAが署名。ルートCAの秘密鍵の露出リスクを軽減
  - サーバー証明書: 中間CAが署名。ドメインに紐づく
  
  証明書の内容:
  - サブジェクト（ドメイン名）、公開鍵、有効期限
  - 発行者（CA）、署名アルゴリズム、デジタル署名
  
  Let's Encrypt:
  - 無料のCA。ACMEプロトコルで自動更新
  - 90日有効期限 → 短い方がセキュリティ上好ましい

----------------------------------------------------------------------
  TLS 1.3
----------------------------------------------------------------------
  この節の狙い: 「TLS 1.3」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  TLS 1.2からの主な改善:
  
  1. ハンドシェイク高速化: 1-RTT（1.2は2-RTT）、0-RTT再接続も可能
  2. 危険な暗号スイートの削除: RC4, DES, CBC, SHA-1等を排除
  3. 前方秘匿性の必須化: 静的RSA鍵交換を廃止、ECDHEのみ
  4. ハンドシェイクの暗号化: ServerHello以降が暗号化される
  
  TLS 1.3のハンドシェイク（簡略版）:
  1. Client → Server: ClientHello + 鍵共有パラメータ + 対応暗号スイート
  2. Server → Client: ServerHello + 鍵共有 + 証明書 + Finished
  3. Client → Server: Finished
  → 1往復で完了。以降はAES-GCMで暗号化通信
  
  許可される暗号スイート（5つのみ）:
  - TLS_AES_128_GCM_SHA256
  - TLS_AES_256_GCM_SHA384
  - TLS_CHACHA20_POLY1305_SHA256 等

  ⚠ よくある誤解: HTTPSなら安全
  → 正確には: TLSは通信経路の暗号化のみ。サーバー側のSQLインジェクションやXSSは防げない。End-to-Endのセキュリティには多層防御が必要

  【面接向けに整理すると】
  Q: TLS 1.3の改善点を説明してください
  A: 主に4点。(1)ハンドシェイクが1-RTTに短縮（0-RTTも可能）、
     (2)脆弱な暗号スイート（CBC, RC4等）を完全排除、
     (3)前方秘匿性を必須化（ECDHE only）、
     (4)ハンドシェイク自体を暗号化しプライバシー向上。
     結果として「速く、安全に、シンプルに」なった。

======================================================================
  Chapter 3: OWASP Top 10 — Webアプリの代表的脆弱性
======================================================================

----------------------------------------------------------------------
  OWASP Top 10 概要（2021年版）
----------------------------------------------------------------------
  先に結論: 「OWASP Top 10 概要（2021年版）」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  OWASPはWebセキュリティのオープンコミュニティ。
  Top 10は「最も重大なWebアプリケーションリスク」のランキング。
  
  A01: Broken Access Control（アクセス制御の不備）
  A02: Cryptographic Failures（暗号化の失敗）
  A03: Injection（インジェクション）
  A04: Insecure Design（安全でない設計）
  A05: Security Misconfiguration（設定ミス）
  A06: Vulnerable Components（脆弱なコンポーネント）
  A07: Identification and Authentication Failures（認証の失敗）
  A08: Software and Data Integrity Failures（整合性の検証不備）
  A09: Security Logging and Monitoring Failures（ログ・監視の不備）
  A10: Server-Side Request Forgery（SSRF）

----------------------------------------------------------------------
  A03: Injection — SQLインジェクション
----------------------------------------------------------------------
  この節の狙い: 「A03: Injection — SQLインジェクション」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  本質: ユーザー入力が「データ」ではなく「命令」として解釈される脆弱性。
  
  例: "SELECT * FROM users WHERE id = '" + userInput + "'"
  入力が "1' OR '1'='1" → 全ユーザー情報が漏洩
  
  対策の優先順位:
  1. パラメータ化クエリ（プリペアドステートメント）→ 最優先
     - SQL文と値を分離。値は常にデータとして扱われる
  2. ストアドプロシージャ（パラメータ化されたもの）
  3. 入力バリデーション（ホワイトリスト方式）
  4. エスケープ処理（最終手段。漏れが生じやすい）
  
  SQLインジェクション以外のInjection:
  - NoSQLインジェクション: MongoDBの$where等
  - LDAPインジェクション
  - OSコマンドインジェクション
  - テンプレートインジェクション（SSTI）

----------------------------------------------------------------------
  A07: XSS — クロスサイトスクリプティング
----------------------------------------------------------------------
  まず全体像: この節では「A07: XSS — クロスサイトスクリプティング」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  攻撃者のJavaScriptがユーザーのブラウザで実行される脆弱性。
  
  3種類:
  - Reflected XSS: URLパラメータ経由。フィッシングメールで誘導
  - Stored XSS: DBに保存された悪意あるスクリプトが他ユーザーに表示
  - DOM-based XSS: クライアント側JSでDOM操作時に発生
  
  対策:
  - 出力エスケープ（コンテキストに応じて: HTML, JS, URL, CSS）
  - Content Security Policy（CSP）ヘッダー
  - HttpOnly Cookie（JSからCookie読み取り不可）
  - フレームワークの自動エスケープ機能を活用（React等）

----------------------------------------------------------------------
  A01: Broken Access Control
----------------------------------------------------------------------
  この節の狙い: 「A01: Broken Access Control」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  認証済みユーザーが、権限のないリソースにアクセスできてしまう問題。
  
  代表的パターン:
  - IDOR（Insecure Direct Object Reference）:
    /api/users/123 のIDを /api/users/456 に変えるだけで他人のデータ取得
  - 権限昇格: 一般ユーザーが管理者APIを直接呼び出せる
  - パストラバーサル: ../../etc/passwd でサーバーのファイルを読める
  
  対策:
  - サーバー側で必ず認可チェック（クライアント側UIの非表示は対策にならない）
  - デフォルト拒否（deny by default）
  - リソースへのアクセス時に所有者チェック
  - 自動テスト（認可のテストケース）

----------------------------------------------------------------------
  A10: SSRF — Server-Side Request Forgery
----------------------------------------------------------------------
  この節の狙い: 「A10: SSRF — Server-Side Request Forgery」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  サーバーに「代わりにリクエストを送らせる」攻撃。
  
  例: 画像URLを指定する機能で http://169.254.169.254/... を指定
  → AWS EC2のメタデータ（IAMクレデンシャル含む）を取得
  
  2019年Capital Oneの大規模情報漏洩はSSRFが起点。
  
  対策:
  - URLのホワイトリスト検証
  - 内部ネットワークへのリクエストをブロック
  - AWS: IMDSv2（トークン必須）を使用
  - DNS Rebinding対策

  ⚠ よくある誤解: WAFを入れればWebアプリの脆弱性は全て防げる
  → 正確には: WAFはシグネチャベースの追加防御層。アプリケーションレベルの根本対策（パラメータ化クエリ、出力エスケープ等）が最優先

  【面接向けに整理すると】
  Q: OWASP Top 10で最も重要と考えるものは？
  A: A01 Broken Access Controlが2021年版で1位。理由は、認証を突破せず
     とも正規ユーザーが権限外のデータにアクセスできるため影響が大きい。
     対策はサーバー側での認可チェック徹底とデフォルト拒否の原則。
     IDORは特に見落とされやすく、APIテストでの網羅的な検証が重要。

======================================================================
  Chapter 4: 認証・認可 — OAuth 2.0, OIDC, JWT, SAML
======================================================================

----------------------------------------------------------------------
  認証と認可の違い
----------------------------------------------------------------------
  先に結論: 「認証と認可の違い」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  認証（Authentication / AuthN）: 「あなたは誰か？」→ 身元の確認
  認可（Authorization / AuthZ）: 「あなたは何ができるか？」→ 権限の確認
  
  アナロジー:
  - 認証 = 社員証で「田中さん」と確認すること
  - 認可 = 田中さんが「会議室Aの予約権限を持つ」と確認すること

----------------------------------------------------------------------
  OAuth 2.0 — 認可のフレームワーク
----------------------------------------------------------------------
  この節の狙い: 「OAuth 2.0 — 認可のフレームワーク」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  目的: 第三者アプリに「パスワードを渡さずに」リソースへのアクセスを許可。
  
  登場人物:
  - Resource Owner: ユーザー（あなた）
  - Client: サードパーティアプリ
  - Authorization Server: 認可を行うサーバー（Google等）
  - Resource Server: 保護されたリソース（Google Drive等）
  
  Authorization Code Flow（最も安全）:
  1. Client → ユーザーをAuthorization Serverへリダイレクト
  2. ユーザーが認可を承認 → Authorization Code発行
  3. Client → Authorization ServerにCodeを送信 → Access Token取得
  4. Client → Resource ServerにAccess Tokenで API呼び出し
  
  PKCE（Proof Key for Code Exchange）:
  - SPAやモバイルアプリ向けの拡張
  - Code Verifier/Challengeで認可コード横取り攻撃を防止
  - 現在は全てのクライアントでPKCE推奨
  
  他のフロー:
  - Client Credentials: サーバー間通信（ユーザー不在）
  - Device Flow: TVやCLIツール用
  - Implicit Flow: 廃止推奨（トークンがURL fragmentに露出）

----------------------------------------------------------------------
  OIDC — OpenID Connect
----------------------------------------------------------------------
  先に結論: 「OIDC — OpenID Connect」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  OAuth 2.0の上に「認証」レイヤーを追加したプロトコル。
  
  OAuth 2.0は「認可」のみ → 「誰がログインしたか」は規定していない。
  OIDCはAccess Tokenに加えてID Tokenを発行。
  
  ID Token:
  - JWT形式
  - iss（発行者）、sub（ユーザー識別子）、aud（対象クライアント）
  - exp（有効期限）、iat（発行日時）
  - 「Googleでログイン」はOIDCの典型的な実装

----------------------------------------------------------------------
  JWT — JSON Web Token
----------------------------------------------------------------------
  先に結論: 「JWT — JSON Web Token」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  構造: header.payload.signature（Base64URLエンコード、ドット区切り）
  
  Header: {"alg": "RS256", "typ": "JWT"}
  Payload: {"sub": "user123", "exp": 1700000000, ...}
  Signature: ヘッダー+ペイロードを秘密鍵で署名
  
  JWS vs JWE:
  - JWS: 署名付き（改ざん検知）。ペイロードは誰でも読める
  - JWE: 暗号化済み。ペイロードも秘匿
  
  JWTのセキュリティ注意点:
  - alg: none攻撃 → アルゴリズムを必ずサーバー側で固定
  - 秘密情報をペイロードに入れない（Base64は暗号化ではない）
  - 有効期限を短く設定（15分〜1時間）
  - Refresh Tokenとの組み合わせでセッション管理

  ⚠ よくある誤解: JWTはセッションCookieの上位互換
  → 正確には: JWTは無効化が困難（ステートレスゆえの弱点）。ログアウトや権限変更の即時反映にはブラックリスト等の仕組みが必要。セッションの方が適切なケースも多い

----------------------------------------------------------------------
  SAML — Security Assertion Markup Language
----------------------------------------------------------------------
  先に結論: 「SAML — Security Assertion Markup Language」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  エンタープライズSSO（シングルサインオン）の標準プロトコル。
  
  OAuth/OIDCとの違い:
  - XML ベース（OAuth/OIDCはJSON）
  - 主にWebブラウザのSSOに特化
  - エンタープライズ（社内システム連携）で広く使用
  - SAML Assertion = 認証情報を含むXML文書
  
  現代の選択:
  - 新規開発 → OIDC推奨（モバイル・API対応、JSONでシンプル）
  - 既存エンタープライズ連携 → SAMLも依然として必要

  【面接向けに整理すると】
  Q: OAuth 2.0とOIDCの違いを説明してください
  A: OAuth 2.0は「認可」のフレームワーク。第三者にリソースアクセスを
     委譲する仕組みで、Access Tokenを発行する。ただし「誰がログインしたか」
     は規定していない。OIDCはOAuth 2.0の上にID Token（JWT形式）による
     「認証」レイヤーを追加したプロトコル。「Googleでログイン」のような
     ソーシャルログインはOIDCの典型例。

======================================================================
  Chapter 5: ゼロトラスト — 信頼しない、常に検証する
======================================================================

----------------------------------------------------------------------
  なぜゼロトラストが必要か
----------------------------------------------------------------------
  この節の狙い: 「なぜゼロトラストが必要か」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  従来のモデル（境界防御）:
  - 社内ネットワーク = 安全、外部 = 危険
  - VPNで社内に入れば全リソースにアクセス可能
  - 「城と堀」モデル
  
  問題点:
  - クラウド移行で「社内」の境界が曖昧に
  - リモートワークの普及
  - 一度侵入されると横移動（Lateral Movement）が容易
  - VPN自体が攻撃対象になる
  
  ゼロトラストの原則:
  1. Never trust, always verify（信頼しない、常に検証）
  2. Least privilege（最小権限）
  3. Assume breach（侵害を前提とする）

----------------------------------------------------------------------
  BeyondCorp — Googleのゼロトラスト実装
----------------------------------------------------------------------
  まず全体像: この節では「BeyondCorp — Googleのゼロトラスト実装」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  2014年にGoogleが発表。VPNを廃止したアーキテクチャ。
  
  核心: アクセス判断を「ネットワーク位置」ではなく
  「ユーザー + デバイス + コンテキスト」で行う。
  
  構成要素:
  - Access Proxy: 全リクエストの認証・認可ゲートウェイ
  - Device Inventory: デバイスの信頼度を管理
  - Access Control Engine: ポリシーに基づく動的判断
  - Trust Level: ユーザー×デバイス×時間×場所で信頼度を算出
  
  例: 社用PCで社内WiFiからアクセス → 高信頼度 → 全サービス利用可
      個人PCでカフェからアクセス → 低信頼度 → メールのみ可

----------------------------------------------------------------------
  mTLS — 相互TLS認証
----------------------------------------------------------------------
  この節の狙い: 「mTLS — 相互TLS認証」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  通常のTLS: サーバーの身元のみ検証（サーバー証明書）
  mTLS: クライアントの身元も検証（クライアント証明書も必要）
  
  用途:
  - マイクロサービス間通信の認証
  - API Gateway → バックエンドサービスの通信
  - ゼロトラスト環境でのサービス間信頼確立
  
  課題:
  - 証明書の管理が煩雑（発行・更新・失効）
  - → サービスメッシュ（Istio等）が自動化

----------------------------------------------------------------------
  SPIFFE / SPIRE — サービスIDの標準
----------------------------------------------------------------------
  この節の狙い: 「SPIFFE / SPIRE — サービスIDの標準」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  SPIFFE（Secure Production Identity Framework For Everyone）:
  - ワークロードに暗号学的なIDを付与する標準仕様
  - SPIFFE ID: spiffe://trust-domain/workload-identifier
  - 人間の認証ではなく「サービス同士」の認証
  
  SVID（SPIFFE Verifiable Identity Document）:
  - X.509証明書またはJWTの形式でIDを表現
  
  SPIRE:
  - SPIFFEの実装。証明書の自動発行・ローテーション
  - Kubernetes、VM、ベアメタルに対応
  
  なぜ重要か:
  - マイクロサービスでは「このサービスは本物の決済サービスか？」を
    暗号学的に証明する必要がある
  - IPアドレスやネットワークセグメントでは不十分

  ⚠ よくある誤解: ゼロトラストは製品を買えば実現できる
  → 正確には: ゼロトラストは「考え方・アーキテクチャ」であり特定製品ではない。ID管理、デバイス管理、ポリシーエンジン、ログ分析等を組み合わせて段階的に実装する

  【面接向けに整理すると】
  Q: ゼロトラストを実現するために最初に取り組むべきことは？
  A: まず「強固なID基盤」の構築。全ユーザー・全サービスに一意のIDを
     付与し、MFA（多要素認証）を必須化する。次にアクセスポリシーの
     定義とAccess Proxyの導入。段階的に進め、VPN依存を徐々に削減する。
     デバイス信頼度の評価とコンテキスト認識も重要な要素。

======================================================================
  Chapter 6: クラウドセキュリティ — IAM・KMS・Secrets管理
======================================================================

----------------------------------------------------------------------
  共有責任モデル
----------------------------------------------------------------------
  先に結論: 「共有責任モデル」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  クラウドセキュリティの大前提: 責任の分担。
  
  AWSの場合:
  - AWSの責任: 物理インフラ、ハイパーバイザー、ネットワーク
  - 顧客の責任: OS設定、アプリケーション、データ、IAM
  
  IaaS → PaaS → SaaS とマネージド度が上がるほど
  クラウド事業者の責任範囲が広がり、顧客の責任は狭まる。
  ただし「データの保護」は常に顧客の責任。

----------------------------------------------------------------------
  IAM — Identity and Access Management
----------------------------------------------------------------------
  先に結論: 「IAM — Identity and Access Management」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  最小権限の原則（Principle of Least Privilege）:
  - 必要最小限の権限のみ付与
  - 「何でもできるAdmin権限」を日常使用しない
  
  AWS IAMのベストプラクティス:
  - ルートアカウントにMFAを設定、日常使用しない
  - IAMユーザーよりIAMロール（一時的な認証情報）を使う
  - ポリシーは明示的なDenyが最優先
  - Service Control Policy（SCP）で組織全体にガードレール
  
  評価順序: 明示的Deny > 明示的Allow > デフォルトDeny
  
  よくある事故:
  - S3バケットのパブリックアクセス設定ミス
  - IAMキーをGitHubにコミット
  - 過剰な権限のLambda実行ロール

----------------------------------------------------------------------
  KMS — 鍵管理サービス
----------------------------------------------------------------------
  この節の狙い: 「KMS — 鍵管理サービス」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  暗号鍵のライフサイクル管理: 生成→使用→ローテーション→廃棄。
  
  AWS KMS:
  - CMK（Customer Master Key）をHSM（Hardware Security Module）で保護
  - エンベロープ暗号化: CMKでデータキーを暗号化、データキーでデータを暗号化
  - 鍵の自動ローテーション（年1回）
  - CloudTrailで鍵の使用履歴を監査
  
  エンベロープ暗号化の理由:
  - 大量データをKMSに直接送るのは非効率（API制限、ネットワーク負荷）
  - データキーをローカルで使い、データキー自体のみKMSで保護
  - 鍵のローテーション時もデータの再暗号化が不要
  
  暗号化の種類:
  - 保存時暗号化（Encryption at Rest）: S3, RDS, EBS等
  - 転送時暗号化（Encryption in Transit）: TLS
  - 使用時暗号化（Confidential Computing）: 新興技術

----------------------------------------------------------------------
  Secrets Manager / Parameter Store
----------------------------------------------------------------------
  先に結論: 「Secrets Manager / Parameter Store」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  秘密情報（DB接続文字列、APIキー、証明書）の安全な管理。
  
  やってはいけないこと:
  - ソースコードにハードコード
  - 環境変数に平文で設定（プロセス一覧で見える場合がある）
  - 設定ファイルをGitリポジトリにコミット
  
  AWS Secrets Manager:
  - 秘密情報の暗号化保存・取得
  - 自動ローテーション（Lambda関数で実現）
  - RDS, Redshift等との統合ローテーション
  - クロスアカウントアクセス対応
  
  GCP Secret Manager / Azure Key Vault も同等の機能。
  
  GitGuardian / git-secrets:
  - コミット時に秘密情報の混入を検知
  - pre-commitフックとして設定

  ⚠ よくある誤解: クラウドに置けばセキュリティはクラウド事業者任せで良い
  → 正確には: 共有責任モデルにより、データ保護・アクセス制御・アプリケーションセキュリティは常に顧客の責任。むしろ設定ミスによる事故が増加している

  【面接向けに整理すると】
  Q: クラウド環境でのシークレット管理のベストプラクティスは？
  A: (1) Secrets Manager等の専用サービスに暗号化保存、
     (2) IAMロールで最小権限のアクセス制御、
     (3) 自動ローテーションの設定、
     (4) pre-commitフックで誤コミット防止、
     (5) CloudTrail等で秘密情報へのアクセスを監査。
     コードに秘密情報を含めない（12-Factor App原則）。

======================================================================
  Chapter 7: コンテナセキュリティ — スキャン・ポリシー・脅威モデリング
======================================================================

----------------------------------------------------------------------
  コンテナの攻撃面
----------------------------------------------------------------------
  先に結論: 「コンテナの攻撃面」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  コンテナはVMほどの分離強度がない。ホストカーネルを共有。
  
  攻撃ベクトル:
  - 脆弱なベースイメージ（古いOS、未パッチのライブラリ）
  - 特権コンテナ（--privileged）での実行
  - Secretsのイメージへのハードコード
  - コンテナエスケープ（カーネル脆弱性を悪用してホストに脱出）
  - サプライチェーン攻撃（悪意あるイメージの利用）

----------------------------------------------------------------------
  イメージスキャン — Trivy
----------------------------------------------------------------------
  先に結論: 「イメージスキャン — Trivy」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  Trivy: Aqua Security製のオープンソーススキャナ。
  
  検出対象:
  - OS パッケージの脆弱性（CVE）
  - アプリケーション依存関係の脆弱性
  - IaCの設定ミス（Dockerfile, Kubernetes YAML, Terraform）
  - 秘密情報のハードコード
  
  CI/CDへの統合:
  - ビルドパイプラインでスキャン → 重大な脆弱性があればデプロイ停止
  - 「Shift Left」: セキュリティチェックを開発の早い段階で実施
  
  Dockerfileのベストプラクティス:
  - マルチステージビルド（最終イメージを最小化）
  - 非rootユーザーで実行（USER命令）
  - distroless / alpine等の最小ベースイメージ
  - .dockerignoreで不要ファイルを除外

----------------------------------------------------------------------
  OPA / Gatekeeper — ポリシーエンジン
----------------------------------------------------------------------
  この節の狙い: 「OPA / Gatekeeper — ポリシーエンジン」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  OPA（Open Policy Agent）: 汎用ポリシーエンジン。
  Rego言語でポリシーを記述。
  
  Gatekeeper: KubernetesにOPAを統合するAdmission Controller。
  
  ポリシー例:
  - 特権コンテナの禁止
  - latest タグの禁止（イメージの再現性確保）
  - 特定レジストリからのイメージのみ許可
  - リソース制限（CPU/Memory）の必須化
  
  宣言的ポリシーの利点:
  - コードとしてGit管理（Policy as Code）
  - テスト可能
  - 監査証跡

----------------------------------------------------------------------
  Pod Security Standards
----------------------------------------------------------------------
  先に結論: 「Pod Security Standards」は実装と設計判断に直結する。背景とトレードオフを押さえる。
  Kubernetes 1.25以降の標準的なPodセキュリティ制御。
  （旧PodSecurityPolicyは廃止）
  
  3つのレベル:
  - Privileged: 制限なし（システムコンポーネント用）
  - Baseline: 最小限の制限（既知の権限昇格を防止）
  - Restricted: 厳格な制限（セキュリティベストプラクティス準拠）
  
  Restrictedの主な制約:
  - 非rootユーザーで実行必須
  - 特権エスカレーション禁止（allowPrivilegeEscalation: false）
  - ホストネットワーク・ホストPID使用禁止
  - 読み取り専用ルートファイルシステム推奨

----------------------------------------------------------------------
  脅威モデリング — STRIDE
----------------------------------------------------------------------
  この節の狙い: 「脅威モデリング — STRIDE」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  設計段階でセキュリティリスクを体系的に分析する手法。
  
  STRIDE（6つの脅威カテゴリ）:
  - Spoofing（なりすまし）: 他者になりすましてアクセス
  - Tampering（改ざん）: データや設定の不正変更
  - Repudiation（否認）: 行為の否定（ログ不足）
  - Information Disclosure（情報漏洩）: 未認可の情報取得
  - Denial of Service（サービス拒否）: 可用性の阻害
  - Elevation of Privilege（権限昇格）: 不正な権限取得
  
  実施手順:
  1. システムのデータフロー図（DFD）を作成
  2. 各コンポーネント・データフローにSTRIDEを適用
  3. リスクの深刻度を評価（DREAD等）
  4. 対策を設計に組み込む
  
  重要: 脅威モデリングは「設計段階」で行うのが最もコスパが良い。
  リリース後の修正は設計段階の6〜15倍のコストがかかる。

  ⚠ よくある誤解: コンテナはサンドボックスなので安全
  → 正確には: コンテナはプロセス分離（namespace/cgroup）であり、VMのようなハードウェアレベルの分離ではない。カーネル脆弱性でエスケープのリスクがあり、多層防御が必要

  【面接向けに整理すると】
  Q: コンテナセキュリティで重要な対策を3つ挙げてください
  A: (1) イメージスキャン（Trivy等）をCI/CDに統合し脆弱性を早期検出、
     (2) 最小権限の原則（非root実行、Pod Security Standards Restricted）、
     (3) ポリシーエンジン（OPA/Gatekeeper）で組織的なルールを強制。
     加えて、最小ベースイメージ使用とSupply Chain Security（署名検証）も重要。

======================================================================
  Chapter 8: インシデント対応 — NIST IR・フォレンジック・SIEM/SOAR
======================================================================

----------------------------------------------------------------------
  NIST インシデント対応フレームワーク
----------------------------------------------------------------------
  この節の狙い: 「NIST インシデント対応フレームワーク」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  NIST SP 800-61: インシデント対応の標準的なフレームワーク。
  
  4つのフェーズ:
  
  1. 準備（Preparation）:
     - インシデント対応チーム（CSIRT）の編成
     - 対応手順書（Playbook）の整備
     - ツール・環境の準備（フォレンジックツール、隔離環境）
     - 訓練・演習の定期実施
  
  2. 検知と分析（Detection & Analysis）:
     - ログ分析、アラート対応
     - インシデントの分類と優先度付け
     - 影響範囲の特定
     - IoC（Indicator of Compromise）の収集
  
  3. 封じ込め・根絶・復旧（Containment, Eradication, Recovery）:
     - 短期封じ込め: 影響拡大の即時阻止（ネットワーク隔離等）
     - 長期封じ込め: 根本原因の除去準備
     - 根絶: マルウェア除去、脆弱性修正
     - 復旧: システム復元、正常動作の確認
  
  4. 事後活動（Post-Incident Activity）:
     - 教訓（Lessons Learned）の文書化
     - 対応手順の改善
     - 経営層・利害関係者への報告
     - 必要に応じて法的対応・通知

----------------------------------------------------------------------
  デジタルフォレンジック
----------------------------------------------------------------------
  この節の狙い: 「デジタルフォレンジック」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  目的: インシデントの証拠を法的に有効な形で保全・分析する。
  
  基本原則:
  - 証拠の完全性: ハッシュ値で改ざんされていないことを証明
  - Chain of Custody: 証拠の取り扱い記録を厳密に管理
  - 揮発性の順序（Order of Volatility）: 消えやすいものから先に収集
    メモリ → ネットワーク接続 → プロセス → ディスク → ログ
  
  クラウドフォレンジックの課題:
  - インスタンスが一時的（スケールイン/アウトで消える）
  - マルチテナント環境での証拠収集の制約
  - 対策: CloudTrail/VPCフローログの事前有効化、EBSスナップショット

----------------------------------------------------------------------
  SIEM — Security Information and Event Management
----------------------------------------------------------------------
  まず全体像: この節では「SIEM — Security Information and Event Management」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  目的: 組織全体のセキュリティログを集約・相関分析・アラート。
  
  機能:
  - ログ集約: サーバー、ネットワーク機器、アプリケーション等
  - 正規化: 異なるフォーマットのログを統一形式に変換
  - 相関分析: 複数ソースのイベントを関連付けて脅威を検知
  - ダッシュボード・レポート: 可視化と監査対応
  
  代表的なプロダクト:
  - Splunk: 最も普及。高機能だが高コスト
  - Elastic Security (ELK): オープンソースベース
  - AWS Security Hub + GuardDuty: AWSネイティブ
  - Microsoft Sentinel: Azure/M365統合
  
  課題:
  - アラート疲れ（Alert Fatigue）: 大量の誤検知でチームが疲弊
  - 対策: チューニング、優先度付け、自動化（SOAR）

----------------------------------------------------------------------
  SOAR — Security Orchestration, Automation and Response
----------------------------------------------------------------------
  この節の狙い: 「SOAR — Security Orchestration, Automation and Response」を単語暗記ではなく、なぜ必要かまでつながる形で理解する。
  目的: インシデント対応の自動化とワークフロー管理。
  
  SIEMとの関係:
  - SIEM = 検知（Detection）
  - SOAR = 対応（Response）の自動化
  - SIEMで検知 → SOARが自動でPlaybook実行
  
  自動化の例:
  - フィッシングメール検知 → 自動で送信者ブロック + 受信者に通知
  - 不審なIPからのアクセス → 自動でファイアウォールルール追加
  - マルウェア検知 → 自動でエンドポイント隔離 + チケット作成
  
  Playbook（対応手順書）:
  - 手動だった対応手順をコード化・自動化
  - 一貫性のある対応を担保
  - 対応時間（MTTR）の短縮

----------------------------------------------------------------------
  セキュリティの指標
----------------------------------------------------------------------
  まず全体像: この節では「セキュリティの指標」を、定義 -> 仕組み -> 実務上の判断の順で整理する。
  重要なKPI:
  - MTTD（Mean Time To Detect）: 侵害の検知までの平均時間
    業界平均: 約200日。目標: できるだけ短く
  - MTTR（Mean Time To Respond）: 検知から対応完了までの時間
  - MTTC（Mean Time To Contain）: 封じ込めまでの時間
  
  セキュリティ成熟度:
  Level 1: 反応的（インシデント発生後に対応）
  Level 2: 管理的（手順書・チーム体制が整備）
  Level 3: 予防的（脅威モデリング、定期スキャン）
  Level 4: 予測的（脅威インテリジェンス、レッドチーム演習）

  ⚠ よくある誤解: SIEMを導入すれば全ての攻撃を検知できる
  → 正確には: SIEMは「ログを集めて分析する」ツール。適切なログソースの設定、検知ルールのチューニング、アナリストによる判断が不可欠。ツールだけでなく人・プロセスの整備が必要

  【面接向けに整理すると】
  Q: インシデント発生時、最初の30分で何をすべきですか？
  A: (1) 影響範囲の初期評価（何が侵害されたか、進行中か）、
     (2) 短期封じ込め（ネットワーク隔離、アカウント無効化等）で拡大防止、
     (3) 証拠の保全（メモリダンプ、ログの確保）、
     (4) エスカレーション（CSIRT、経営層、必要に応じて法務・広報）。
     パニックせず、手順書に従い記録を残しながら対応することが重要。
