----------------------------------------------------------------------
  OKR とは何か
----------------------------------------------------------------------
  OKR は Objectives and Key Results の略です。
  Google が採用して広まった目標設定フレームワークです。
  
  Objective (目標):
  - 定性的で、チームを鼓舞する方向性
  - 例: 「エンジニアの技術的意思決定を加速する」
  
  Key Results (主要成果指標): 
  ード。
  
  Signature (署名):
  RSASHA256(
    base64UrlEncode(header) + "." + base64UrlEncode(payload),
    秘密鍵
  )
  
  なぜ Base64URL か:
  JSON にはスペースや特殊文字が含まれるため、
  URL やヘッダに安全に含められる形式にエンコードします。
  Base64URL は Base64 から + を - に、/ を _ に置換したものです。
  
  重要: Payload は暗号化されていない
  Base64 はエンコード (可逆変換) であって暗号化ではありません。
  誰でもデコードして中身を読めます。
  だから機密情報 (パスワード等) を入れてはいけません。
  署名は「改ざんされていないこと」を検証するだけです。

----------------------------------------------------------------------
  OAuth 2.0
----------------------------------------------------------------------
  OAuth 2.0 は「認可 (Authorization)」のプロトコルです。
  認証 (Authentication: あなたは誰？) とは異なり、
  認可は「あなたは何ができるか？」を扱います。
  
  よくある使い方:
  「Google でログイン」ボタンを押すと、
  Google があなたの代わりに TechPulse に
  「この人のメールアドレスとプロフィール画像」へのアクセスを許可します。
  
  Authorization Code Flow (最も安全):
  
  1. ユーザーがTechPulseの「Googleでログイン」を押す
  2. TechPulseがGoogleの認可エンドポイントにリダイレクト
  3. ユーザーがGoogleでログイン → 許可する
  4. GoogleがTechPulseに「認可コード」を返す (リダイレクト)
  5. TechPulseのサーバーが「認可コード」を使って
     Googleのトークンエンドポイントに「アクセストークン」を要求
  6. アクセストークンで Google API からユーザー情報を取得
  
  なぜ認可コードを挟むか:
  アクセストークンをブラウザのリダイレクトURL に直接含めると、
  ブラウザの履歴やログに残ってしまうリスクがあります。
  認可コードはサーバー間通信でアクセストークンと交換するため、
  トークンがブラウザに露出しません。

----------------------------------------------------------------------
  RSA の数学的原理
----------------------------------------------------------------------
  JWT の署名アルゴリズム RS256 は RSA を使います。
  RSA がなぜ安全なのか、数学的に理解しましょう。
  
  基本原理: 素因数分解の困難性
  
  大きな素数 p と q を掛けるのは簡単:
    p = 61, q = 53 → n = p × q = 3233
  
  しかし n = 3233 から p, q を見つけるのは
  (小さい数なら簡単ですが) n が数百桁になると現実的に不可能です。
  
  鍵の生成:
  1. 大きな素数 p, q を選ぶ (それぞれ 1024 ビット以上)
  2. n = p × q を計算
  3. φ(n) = (p-1)(q-1) を計算 (オイラーのトーシェント関数)
  4. e を選ぶ (通常 65537)。φ(n) と互いに素であること
  5. d を計算: e × d ≡ 1 (mod φ(n))
  
  公開鍵: (n, e)
  秘密鍵: (n, d)
  
  署名:
  sig = message^d mod n  (秘密鍵で署名)
  
  検証:
  message = sig^e mod n  (公開鍵で検証)
  
  なぜ安全か:
  秘密鍵 d を知るには φ(n) = (p-1)(q-1) が必要。
  φ(n) を知るには n を素因数分解して p, q を求める必要がある。
  n が十分大きければ素因数分解は計算量的に不可能。

----------------------------------------------------------------------
  レート制限
----------------------------------------------------------------------
  API を保護するために、単位時間あたりのリクエスト数を制限します。
  
  Token Bucket アルゴリズム:
  バケツにトークンが入っている。
  リクエストが来るたびにトークンを1つ消費する。
  一定間隔でトークンが補充される。
  バケツが空ならリクエストを拒否 (429 Too Many Requests)。
  
  特徴:
  - バースト許容: バケツにトークンが溜まっていれば
    短期間に多くのリクエストを受け付ける
  - 実装が簡単
  
  Sliding Window:
  過去 N 秒間のリクエスト数をカウントし、
  閾値を超えたら拒否する。
  
  特徴:
  - バースト耐性: 固定ウィンドウの境界でのバーストを防ぐ
  - Token Bucket より実装がやや複雑

----------------------------------------------------------------------
  SQL Injection と防御
----------------------------------------------------------------------
  SQL Injection は OWASP Top 10 の A03 に位置する深刻な脆弱性です。
  
  攻撃例:
  ユーザーが検索ボックスに以下を入力:
    ' OR '1'='1
  
  アプリケーションが文字列結合でSQLを組み立てている場合:
    SELECT * FROM posts WHERE title = '' OR '1'='1'
  
  '1'='1' は常に true なので、全投稿が返ってしまいます。
  さらに悪意のある入力:
    '; DROP TABLE posts; --
  
  これでテーブルが削除される可能性があります。
  
  防御: Prepared Statement (パラメータ化クエリ)
  SQL 文とデータを分離します:
    SELECT * FROM posts WHERE title = ?
  データベースエンジンは ? を「SQLの構文」ではなく
  「データ」として扱うため、SQL として解釈されません。
  
  なぜこれで防げるか:
  Prepared Statement では、SQL の構文解析が先に行われ、
  後からバインドされた値はリテラル (文字列データ) として処理されます。
  攻撃者が何を入力しても、それは SQL 命令にはなりません。

  【面接向けに整理すると】
  「API設計で最も重要なことは？」
  
  「リソース指向の設計、適切なHTTPメソッドとステータスコードの使用、
  そしてセキュリティです。具体的にはPrepared Statementによるインジェクション防止、
  JWTによる認証、レート制限による過負荷防止を実装します。
  RESTかGraphQLかgRPCかはユースケースで選びます。」

----------------------------------------------------------------------
  正規化 - なぜテーブルを分けるのか
----------------------------------------------------------------------
  データベース設計の基本は「正規化」です。
  データの重複を排除し、更新時の不整合を防ぎます。
  
  第 1 正規形 (1NF):
  - 各列の値がアトミック (分割不可能) であること
  - 悪い例: tags 列に "Python, Go, Rust" とカンマ区切りで入れる
  - 良い例: post_tags テーブルを別に作り、1行1タグにする
  
  第 2 正規形 (2NF):
  - 1NF を満たし、かつ
  - 非キー列が複合主キーの一部にのみ依存しないこと
  - 例: (post_id, tag_id, tag_name) → tag_name は tag_id だけで決まる
    → tag_name は tags テーブルに分離すべき
  
  第 3 正規形 (3NF):
  - 2NF を満たし、かつ
  - 非キー列が他の非キー列に依存しないこと (推移的依存の排除)
  - 例: (user_id, department_id, department_name)
    → department_name は department_id で決まる
    → departments テーブルに分離すべき
  
  なぜ正規化するか:
  1. データの冗長性排除 → ストレージ節約
  2. 更新異常の防止 → department_name を変えるとき1箇所だけ変更すればいい
  3. データの整合性保証
  
  非正規化する場合:
  読み取り性能が重要な場合、あえて冗長データを持つこともあります。
  例: フィードの読み込みが遅いので、ユーザー名を投稿テーブルにも持つ。
  ただし、ユーザー名変更時に複数箇所の更新が必要になるトレードオフがあります。

----------------------------------------------------------------------
  B+Tree インデックス
----------------------------------------------------------------------
  PostgreSQL がデータを高速に検索するために使うのが B+Tree インデックスです。
  
  B-Tree とは:
  ノードに複数のキーを持つ自己平衡探索木です。
  各ノードが最大 M 個のキーを持つ場合、
  子ノードは最大 M+1 個あります。
  
  B+Tree は B-Tree の改良版:
  1. データはリーフノード (末端) にのみ格納
  2. 内部ノードはキーのコピーだけ持つ (ナビゲーション用)
  3. リーフノード同士がリンクリストで繋がっている
  
  なぜ B+Tree か:
  1. ディスク I/O を最小化:
     1ノードを1ページ (通常 8KB) に収めると、
     1回のディスク読み込みで数百のキーを比較できる。
     バイナリツリーだと1ノード1キーで、深さが大きくなり I/O 回数が増える。
  
  2. 範囲検索が高速:
     リーフのリンクリストを辿るだけで済む。
     例: WHERE created_at BETWEEN '2024-01-01' AND '2024-12-31'
     → 開始位置を見つけたら、リーフを順に辿るだけ。
  
  3. 安定した性能:
     全リーフの深さが同じなので、どのキーにアクセスしても
     同じ回数のディスク I/O で到達できる。

----------------------------------------------------------------------
  Write Path: INSERT からディスクまで
----------------------------------------------------------------------
  投稿が作成されました。INSERT 文が実行されます。
  データがディスクに書き込まれるまでの道のりを追いましょう。
  
  1. SQL パーサー:
     INSERT INTO posts (title, content) VALUES ('Kafka入門', '...')
     → SQL を解析して実行計画を作成
  
  2. WAL (Write-Ahead Log) への書き込み:
     まず「何をしようとしているか」をログに記録します。
     これが WAL です。
  
     なぜデータ本体より先にログを書くか:
     書き込みの途中でクラッシュした場合、
     WAL があれば「どこまで完了していたか」が分かります。
     WAL がなければ、データが中途半端な状態で壊れる可能性があります。
     これが WAL の名前の由来: Write-Ahead (事前に書く) Log
  
  3. Buffer Pool への書き込み:
     実際のテーブルデータはメモリ上の Buffer Pool に書き込みます。
     Buffer Pool はディスク上のページのキャッシュです。
  
     なぜすぐディスクに書かないか:
     ディスク I/O は遅いからです。
     メモリに書いて「後でまとめてディスクに書く」ほうが効率的。
     この時点ではメモリ上のページは「汚れた (dirty)」状態です。
  
  4. Checkpoint (チェックポイント):
     定期的に、dirty ページをディスクにフラッシュします。
     チェックポイント完了後、そこまでの WAL は不要になります。
  
  5. fsync:
     OS の write() はカーネルのページキャッシュに書くだけで、
     まだディスクの永続記憶に到達していません。
     fsync を呼ぶことで、OS がページキャッシュの内容を
     ディスクに書き出すことを保証します。

----------------------------------------------------------------------
  SSD の内部 - FTL
----------------------------------------------------------------------
  fsync でデータがディスク (SSD) に書き込まれます。
  SSD の内部は HDD とは全く異なる仕組みです。
  
  NAND Flash の特性:
  - 読み取り: ページ単位 (4KB)
  - 書き込み: ページ単位 (4KB)
  - 削除: ブロック単位 (256KB〜1MB) ← これが重要
  
  つまり、1ページだけ書き換えたくても、
  ブロック全体を消してから書き直す必要があります。
  
  FTL (Flash Translation Layer):
  SSD のファームウェアが提供する変換層です。
  OS からは通常のブロックデバイスに見えますが、
  内部では以下を行っています:
  
  1. 論理アドレス → 物理アドレスの変換
  2. ウェアレベリング: 書き込みを均等に分散 (セルの寿命を延ばす)
  3. ガベージコレクション: 不要なブロックを回収
  
  書き込み増幅 (Write Amplification):
  OS が 4KB 書き込んでも、SSD 内部では GC のために
  ブロック全体の読み取り→書き戻しが発生し、
  実際の NAND 書き込み量がOS要求量の数倍になることがあります。

----------------------------------------------------------------------
  クエリプランナ - どうやって検索するか
----------------------------------------------------------------------
  SELECT でデータを読む場合、PostgreSQL のクエリプランナが
  「どうやって検索するか」を決めます。
  
  EXPLAIN ANALYZE をつけると実行計画が見られます。
  
  アクセス方式:
  
  Seq Scan (Sequential Scan):
  - テーブルの全行を順番にスキャンする
  - いつ使われるか: 検索条件が広い場合 (全体の20%以上を読む場合)
  - 小さいテーブルでは Index Scan より速いこともある
  
  Index Scan:
  - B+Tree インデックスを辿って該当行を見つける
  - いつ使われるか: 選択性が高い場合 (少数の行を返す)
  - インデックスとテーブルの両方を読むため、
    返す行数が多いと Seq Scan より遅くなる
  
  Bitmap Index Scan:
  - インデックスを使って「該当するページ番号」のビットマップを作り、
    まとめてテーブルを読む
  - 複数のインデックスを AND/OR で組み合わせられる
  
  JOIN の実行方式:
  
  Nested Loop Join:
  - 外側テーブルの各行に対して、内側テーブルを検索
  - 内側にインデックスがあれば高速
  - 小テーブル × 大テーブル (インデックスあり) に最適
  
  Hash Join:
  - 小さいテーブルからハッシュ表を構築
  - 大きいテーブルを走査して、ハッシュ表で照合
  - 等値結合 (= で結合) の場合に非常に高速
  - メモリにハッシュ表が収まる必要がある
  
  Merge Join:
  - 両テーブルを結合キーでソートしてから突き合わせ
  - ソート済みのデータ (インデックスがある場合) に有効
  - 大量データの等値結合に適する

----------------------------------------------------------------------
  Redis Cache-Aside パターン
----------------------------------------------------------------------
  フィードの読み込みが遅い。毎回 JOIN を走らせるのは重い。
  Redis でキャッシュを入れます。
  
  Cache-Aside (Lazy Loading):
  1. アプリが Redis にキーを問い合わせる
  2. キャッシュヒット → Redis から返す (高速)
  3. キャッシュミス → DB から読み、Redis に書き込んでから返す
  
  なぜ Cache-Aside か:
  - アプリが明示的にキャッシュを制御するため、予測しやすい
  - キャッシュに載るのは実際に読まれたデータだけ
  
  問題と対策:
  
  Cache Stampede (雷群問題):
  人気記事のキャッシュが期限切れ → 同時に100リクエストが DB に殺到
  対策: ロックをかけて1つだけ DB に読みに行く (singleflight パターン)
  
  Cache Penetration (貫通):
  存在しないデータへのリクエストが繰り返される
  → 毎回キャッシュミスして DB に問い合わせる
  対策: 存在しないことも短時間キャッシュする (null キャッシュ)
  対策: ブルームフィルタで存在確認
  
  Cache Breakdown (崩壊):
  ホットキーのキャッシュが切れた瞬間に大量リクエスト
  対策: ホットキーには有効期限を設定しない、
        またはバックグラウンドで更新する

----------------------------------------------------------------------
  CAP 定理
----------------------------------------------------------------------
  分散データベースに関する有名な定理です。
  
  3 つの性質のうち、同時に 2 つまでしか保証できない:
  
  C (Consistency): 全ノードが同じデータを返す
  A (Availability): 全リクエストに応答を返す
  P (Partition Tolerance): ネットワーク分断に耐える
  
  なぜ 3 つ同時は無理か:
  ネットワーク分断 (P) が発生した場合を考えます。
  ノード A とノード B が通信できなくなりました。
  
  - C を優先: B がデータを返すと A と不一致かもしれない → B はエラーを返す
    → Availability を犠牲にしている
  - A を優先: B は持っているデータで応答する (古いかもしれない)
    → Consistency を犠牲にしている
  
  分断が起きている間は、C と A のどちらかを選ぶしかありません。
  
  実際には P (分断耐性) は避けられない:
  ネットワークは必ず故障し得ます。
  だから実質的な選択は「CP (一貫性優先)」か「AP (可用性優先)」です。
  
  CP の例: PostgreSQL, HBase, MongoDB (with majority write concern)
  AP の例: Cassandra, DynamoDB, CouchDB

  ⚠ よくある誤解: CAP定理は3つから2つを選ぶ
  → 正確には: Partition (分断) は避けられないので、実質は分断時に Consistency を取るか Availability を取るかの選択。分断がない通常時は C と A の両方を実現できる

----------------------------------------------------------------------
  ACID と分離レベル
----------------------------------------------------------------------
  トランザクションが保証する 4 つの性質:
  
  Atomicity (原子性):
  トランザクション内の操作は「全て成功」か「全て失敗」。
  中途半端な状態にならない。
  例: 送金で「引き落とし」が成功して「入金」が失敗 → 両方ロールバック
  
  Consistency (一貫性):
  トランザクション前後で、DB が整合性制約を満たす。
  例: 残高がマイナスにならない制約
  
  Isolation (分離性):
  並行するトランザクションが互いに影響しない。
  ただし、完全な分離は性能が低下するため、分離レベルを選べる。
  
  Durability (永続性):
  コミットされたデータは永続的に保存される (電源断でも失われない)。
  WAL + fsync がこれを保証する。

----------------------------------------------------------------------
  分離レベル
----------------------------------------------------------------------
  Isolation は性能とのトレードオフがあるため、
  4 段階の分離レベルが定義されています:
  
  Read Uncommitted:
  他のトランザクションのコミット前のデータが見える (dirty read)。
  ほぼ使われない。
  
  Read Committed (PostgreSQL デフォルト):
  コミット済みのデータだけ見える。
  ただし、同じクエリを2回実行すると結果が変わり得る (non-repeatable read)。
  
  Repeatable Read:
  トランザクション開始時点のスナップショットを見る。
  同じクエリは同じ結果を返す。
  ただし、他のトランザクションが INSERT した行が
  見えたり見えなかったりする (phantom read) 問題が理論上ある。
  
  Serializable:
  全トランザクションが順番に実行されたかのように振る舞う。
  最も強い保証だが、性能ペナルティが大きい。

----------------------------------------------------------------------
  MVCC (Multi-Version Concurrency Control)
----------------------------------------------------------------------
  PostgreSQL が Read Committed や Repeatable Read を実現する仕組みです。
  
  基本原理:
  行を更新するとき、古い行を消さずに新しいバージョンを作ります。
  各トランザクションは自分が「見るべきバージョン」を判断します。
  
  PostgreSQL の実装:
  各行に以下のメタデータがある:
  - xmin: この行を作成したトランザクション ID
  - xmax: この行を削除/更新したトランザクション ID (0 = まだ有効)
  
  可視性判定:
  トランザクション TX100 が行を読むとき:
  - xmin < 100 かつコミット済み → 見える (自分より前に作られた)
  - xmin >= 100 → 見えない (自分の後に作られた)
  - xmax > 0 かつ xmax < 100 かつコミット済み → 見えない (削除済み)
  
  MVCC の利点:
  読み取りがロックを取らない。
  書き込みトランザクションがあっても、読み取りはブロックされない。
  
  MVCC の問題:
  古いバージョンの行が溜まる → VACUUM で回収する必要がある。
  VACUUM が遅延すると、テーブルが膨張する (テーブル bloat)。
  autovacuum の設定が重要。

  【面接向けに整理すると】
  「DBの書き込みパスを説明せよ」
  
  「INSERTが実行されると、まずWAL (Write-Ahead Log) に変更内容を記録し、
  次にBuffer Pool上のページを更新します。WALはfsyncで永続化されます。
  Buffer Poolのdirtyページは定期的なCheckpointでディスクにフラッシュされます。
  この設計により、クラッシュ時もWALからリカバリ可能で、
  通常の書き込みはメモリ上で完結するため高速です。
  SSD内部ではFTLが論理→物理アドレス変換やウェアレベリングを行い、
  書き込み増幅を最小化しています。」

======================================================================
  Chapter 4: スケールする - 分散システム (深掘り)
======================================================================

----------------------------------------------------------------------
  Consistent Hashing - なぜ普通のハッシュではダメか
----------------------------------------------------------------------
  ユーザーが100万人を突破しました。1台のDBでは無理です。
  データを複数のサーバーに分散させる必要があります。
  
  単純なアプローチ: hash(key) % N
  キーのハッシュ値を N (サーバー数) で割った余りでサーバーを決める。
  
  問題: サーバーを追加/削除すると N が変わる
  → ほぼ全てのキーの割り当て先が変わる
  → 大量のデータ移動が発生する
  
  例: サーバー3台 → 4台にすると、約75%のキーが移動する。
  
  Consistent Hashing の解決策:
  ハッシュ空間を「リング」として扱う (0 〜 2^32-1)。
  サーバーもキーも、同じハッシュ関数でリング上の位置を決める。
  キーは「時計回りで最初に見つかるサーバー」に割り当てる。
  
  サーバー追加時:
  リング上に新しいサーバーが1つ加わる。
  影響を受けるのは、そのサーバーと「反時計回りの隣のサーバー」の
  間にあるキーだけ。全体の 1/N 程度しか移動しない。
  
  仮想ノード:
  物理サーバーが少ないと、リング上の分布が偏ることがある。
  1つの物理サーバーに複数の仮想的な点をリング上に配置すると、
  データが均等に分散される。
  例: 物理3台 × 仮想150点 = リング上に450点で均等分散。
  
  使われている場所:
  - DynamoDB, Cassandra: データ分散
  - Redis Cluster: ハッシュスロット (16384スロット)
  - CDN: リクエストのルーティング

----------------------------------------------------------------------
  レプリケーション - なぜコピーを持つか
----------------------------------------------------------------------
  シャーディングでデータを分散しましたが、
  1台が壊れたらそのデータが失われます。
  
  レプリケーションはデータのコピーを複数サーバーに持つ仕組みです。
  
  目的:
  1. 可用性: 1台が落ちても別のコピーでサービス継続
  2. 読み取り分散: 読み取りをレプリカに分散して負荷軽減
  3. 地理的分散: ユーザーに近いレプリカから読む
  
  Primary-Replica 方式:
  - Primary (1台): 書き込みを受け付ける
  - Replica (複数台): Primary のデータをコピーして持つ
  
  同期レプリケーション:
  - Primary が書き込みを受けたら、全 Replica に反映されるまで待つ
  - 利点: 常に一貫性がある
  - 問題: 1台の Replica が遅いと全体が遅くなる
  
  非同期レプリケーション:
  - Primary は書き込みをすぐ完了させ、Replica への反映は後で行う
  - 利点: Primary が高速
  - 問題: Replica が最新データに追いつくまでの間、
    古いデータ (stale read) を返す可能性がある
  
  半同期 (Semi-synchronous):
  - 少なくとも1台の Replica に反映されたら完了とする
  - 同期と非同期の中間

----------------------------------------------------------------------
  Raft 合意プロトコル
----------------------------------------------------------------------
  複数のサーバーが「1つの真実」について合意する必要がある場合があります。
  
  例: 「この書き込みはコミット済みか？」
  3台のサーバーのうち2台が「はい」で1台が「いいえ」なら、どうするか？
  
  これが合意問題です。Raft はこれを解決するプロトコルです。
  
  Raft の3つの要素:
  
  1. Leader Election (リーダー選出):
  - 全ノードは Follower として開始
  - 一定時間 Leader からの通信がないと、Candidate になり選挙を開始
  - 過半数の投票を得たら Leader になる
  - Leader は任期 (term) を持ち、任期が終わると再選挙
  
  なぜ1人のリーダーが必要か:
  全ノードが勝手に書き込みを受け付けると、
  順序が決まらず矛盾が生じる。
  1人のリーダーが「この順番で書き込む」と決めることで、
  全ノードが同じ順序でログを持てる。
  
  2. Log Replication (ログ複製):
  - クライアントの書き込みリクエストは Leader が受ける
  - Leader はログエントリを作り、AppendEntries RPC で全 Follower に送信
  - 過半数が記録したら「コミット済み」とする
  
  なぜ「過半数」か:
  5台中3台が同意すれば、2台が落ちても過半数が残る。
  任意の2つの過半数は必ず1台以上が重なるため、
  コミット済みの情報が失われることはない。
  (数学的に: C(5,3) のどの2つの組み合わせも共通要素を持つ)
  
  3. Safety (安全性):
  - コミットされたエントリは取り消せない
  - 新しい Leader は必ず過去のコミット済みエントリを全て持つ

----------------------------------------------------------------------
  Paxos との違い
----------------------------------------------------------------------
  Paxos は Raft より前に発明された合意プロトコルです。
  
  Paxos の2フェーズ:
  Phase 1 (Prepare): Proposer が「提案番号 N で提案したい」と Acceptor に問う
  Phase 2 (Accept): Acceptor が OK なら、Proposer が値を送って合意
  
  Paxos の問題:
  - 理論的に正しいが、実装が非常に複雑
  - 論文自体が難解 (著者の Lamport も認めている)
  - Multi-Paxos の最適化が複雑
  
  Raft の設計思想:
  - 「理解しやすさ」を明示的な設計目標にした
  - Leader ベースで直感的
  - 実装の参考になる擬似コードが論文に含まれている
  
  理論的な能力は同等ですが、実装しやすさで Raft が広く採用されています。
  etcd, CockroachDB, TiDB などが Raft を使っています。

----------------------------------------------------------------------
  FLP 不可能性定理
----------------------------------------------------------------------
  1985年に Fisher, Lynch, Paterson が証明した定理です。
  
  内容:
  「非同期分散システムにおいて、1台でも故障する可能性があれば、
   決定的な合意アルゴリズムは存在しない」
  
  これが意味すること:
  - 非同期: メッセージの到着時間に上限がない
  - 決定的: 必ず有限時間内に結果を出す
  → この条件下では「必ず合意に至る」アルゴリズムは不可能
  
  Raft はどう回避するか:
  タイムアウトを使います。
  一定時間応答がなければ「そのノードは故障した」と判断する。
  これは「同期の仮定」を部分的に導入していることになり、
  FLP の「非同期」前提を緩和しています。
  
  実用的には「いつかは」合意に至る (活性は確率的)。

----------------------------------------------------------------------
  一貫性モデルのスペクトラム
----------------------------------------------------------------------
  分散システムで「どこまでの正しさを保証するか」は設計判断です。
  
  線形化可能性 (Linearizability):
  - 最も強い保証
  - 全操作にグローバルな順序がある
  - 書き込みが完了したら、全ノードで即座に見える
  - 実装: 全操作を Leader 経由にする → 性能ペナルティ大
  - 用途: 銀行残高、在庫管理 (絶対に矛盾不可)
  
  因果一貫性 (Causal Consistency):
  - 因果関係のある操作は順序が保証される
  - 「AさんがBさんのコメントに返信」→ 全ノードでAの後にBの返信が見える
  - 因果関係のない操作は順不同でOK
  - 実装: Vector Clock で因果関係を追跡
  - 用途: SNS のコメント、チャット
  
  セッション保証 (Session Consistency):
  - 同一セッション内では自分の書き込みが読める (Read Your Writes)
  - 他ユーザーの書き込みは遅延あり
  - 用途: TechPulse のフィード (自分の投稿は即座に見える)
  
  結果整合性 (Eventual Consistency):
  - 最も弱い保証
  - いつかは全ノードが同じ値を返す (通常は数百ms〜数秒)
  - 実装: 非同期レプリケーション → 性能は最高
  - 用途: いいね数、閲覧カウンター (多少ズレてもOK)

----------------------------------------------------------------------
  Kafka - メッセージングの仕組み
----------------------------------------------------------------------
  なぜメッセージキューが必要か:
  
  投稿が作成されたら、以下が必要:
  1. フォロワーのフィード更新
  2. 検索インデックス更新
  3. AI モデレーション
  4. 通知送信
  
  これを全て同期的にやると:
  - 投稿 API のレスポンスが遅くなる (全部終わるまで待つ)
  - 1つが失敗すると全部失敗する
  - サービス間の結合度が高い
  
  Kafka を使うと:
  - 投稿サービスは Kafka にイベントを書いたらすぐ返す (非同期)
  - 各サービスが独立してイベントを消費する (疎結合)
  - 一時的に消費者がダウンしてもイベントは Kafka に保持される (バッファ)
  
  Kafka の内部構造:
  
  Topic: メッセージのカテゴリ (例: "post-created")
  Partition: Topic を分割したもの。各 Partition は順序付きログ。
  Offset: Partition 内のメッセージの連番。
  Consumer Group: 複数の Consumer が Partition を分担して読む。
  
  Fan-out on Write vs Fan-out on Read:
  
  Fan-out on Write (Push):
  - 投稿時にフォロワー全員のフィードに書き込む
  - 読み取りは自分のフィードを読むだけ (高速)
  - 問題: フォロワーが100万人いると書き込みが重い
  
  Fan-out on Read (Pull):
  - フィード読み取り時にフォロー中全員の投稿を集める
  - 書き込みは1回だけ (高速)
  - 問題: 読み取りが遅い
  
  ハイブリッド (Twitter方式):
  - 一般ユーザーは Fan-out on Write
  - フォロワーが極端に多い有名人は Fan-out on Read

----------------------------------------------------------------------
  配信セマンティクス
----------------------------------------------------------------------
  メッセージが「ちょうど1回」届くことを保証するのは意外と難しいです。
  
  At-most-once (最大1回):
  - 送信したら確認しない。ロスの可能性あり。
  - 例: UDP でのログ送信
  
  At-least-once (最低1回):
  - ACK が来なければ再送。重複の可能性あり。
  - 例: Kafka のデフォルト設定
  
  Exactly-once (ちょうど1回):
  - 重複なし、ロスなし。最も難しい。
  - Kafka では: 冪等 Producer + トランザクション Consumer で実現
  
  冪等性 (Idempotency) とは:
  同じ操作を何回実行しても結果が変わらないこと。
  例: 「残高を100にセット」は冪等 (何回やっても100)
  例: 「残高に100を加算」は冪等ではない (やるたびに増える)
  
  At-least-once + 冪等な処理 = 実質的に Exactly-once
  これが実務で最も現実的なパターンです。

----------------------------------------------------------------------
  Little's Law と Amdahl's Law
----------------------------------------------------------------------
  Little's Law:
  L = λ × W
  
  L: システム内の平均リクエスト数
  λ: 到着率 (リクエスト/秒)
  W: 平均滞在時間 (レイテンシ)
  
  直感的理解:
  レストランで考えると、
  「店内の客数 = 来店頻度 × 平均滞在時間」
  
  応用: サーバー何台必要か？
  1秒に1000リクエスト (λ=1000) が来て、
  各リクエストの処理に 0.1秒 (W=0.1) かかるなら、
  L = 1000 × 0.1 = 100 リクエストが常にシステム内にある。
  1サーバーが10並列処理できるなら、10台必要。
  
  Amdahl's Law:
  Speedup = 1 / ((1-p) + p/s)
  
  p: 並列化可能な割合
  s: 並列度 (コア数)
  
  重要な洞察:
  並列化できない部分 (1-p) がボトルネックになる。
  処理の90%が並列化可能 (p=0.9) でも、
  10コアにすると Speedup = 1/(0.1 + 0.9/10) = 5.26 倍
  100コアにしても Speedup = 1/(0.1 + 0.9/100) = 9.17 倍
  → 10%の逐次部分が上限を決めてしまう

  【面接向けに整理すると】
  「分散システムで最も難しいことは？」
  
  「一貫性と可用性のトレードオフです。CAP定理により、
  ネットワーク分断時にはどちらかを犠牲にする必要があります。
  TechPulseでは、いいね数には結果整合性、
  ユーザー認証情報には強い一貫性を適用し、
  データの性質に応じて一貫性モデルを使い分けています。
  合意にはRaftプロトコルを採用し、
  過半数コミットにより安全性を保証しています。」

======================================================================
  Chapter 5: AIを組み込む (深掘り)
======================================================================

----------------------------------------------------------------------
  RNN から Transformer へ
----------------------------------------------------------------------
  なぜ Transformer を理解する必要があるか:
  TechPulse の AI 機能 (要約、モデレーション、推薦) は
  全て Transformer ベースの LLM を使っています。
  
  RNN (Recurrent Neural Network) とは:
  テキストを1トークンずつ順番に処理するニューラルネットワークです。
  各ステップで「隠れ状態」を次のステップに渡します。
  
  イメージ:
  「私 は 猫 が 好き」を処理する場合:
  「私」→ 隠れ状態1 → 「は」→ 隠れ状態2 → 「猫」→ ...
  
  RNN の問題:
  1. 逐次処理: 「猫」を処理するには「は」の処理が終わるのを待つ必要がある
     → GPU の並列計算能力を活かせない
  2. 勾配消失: 長い文で最初の方の情報が失われやすい
     (LSTM や GRU で緩和されたが、根本的解決ではない)
  
  Transformer の革新:
  「全トークンを同時に見る」ことで並列処理を可能にした。
  これが Self-Attention メカニズムです。

----------------------------------------------------------------------
  Self-Attention の仕組み
----------------------------------------------------------------------
  Self-Attention は「各トークンが他の全トークンとの関連度を計算する」仕組みです。
  
  Q (Query), K (Key), V (Value):
  入力の各トークンから、3つのベクトルを計算します。
  
  直感的理解:
  図書館で考えると:
  - Query: 「私が探しているもの」
  - Key: 「各本のタイトル (検索キー)」
  - Value: 「各本の中身」
  
  1. Q と K の内積を計算 → 関連度スコア (どの本が関係ありそうか)
  2. スコアを √d_k で割る
  3. Softmax で確率分布にする (合計が1になる)
  4. この確率で V を重み付き平均する (関連する本の内容を混ぜる)
  
  なぜ √d_k で割るか:
  d_k (キーの次元数) が大きいと、内積の値が大きくなりすぎます。
  大きすぎる値に Softmax を適用すると、
  勾配がほぼ0になり学習が進まなくなります (勾配消失)。
  √d_k で割ることでスケールを適切に保ちます。
  
  Multi-Head Attention:
  1つの Attention だけでなく、複数の「ヘッド」で別々に Attention を計算し、
  結果を結合します。
  各ヘッドは「異なる関係性」を捉えます:
  - ヘッド1: 文法的な関係 (主語-動詞)
  - ヘッド2: 意味的な関係 (同義語)
  - ヘッド3: 位置的な関係 (近くの単語)

----------------------------------------------------------------------
  Positional Encoding
----------------------------------------------------------------------
  Transformer は全トークンを同時に処理するため、
  「どのトークンが何番目か」の位置情報を持ちません。
  
  RNN は逐次処理なので自然に位置情報を持っていましたが、
  Transformer は明示的に位置を教える必要があります。
  
  sin/cos による Positional Encoding:
  各位置に、sin と cos 関数で計算したベクトルを割り当てます。
  周波数を変えることで、近い位置同士は似たベクトル、
  遠い位置同士は異なるベクトルになります。
  
  このベクトルを Embedding (単語のベクトル表現) に加算することで、
  モデルが位置情報を活用できるようになります。

----------------------------------------------------------------------
  推薦システム
----------------------------------------------------------------------
  TechPulse のフィードには、ユーザーに合った記事を表示したい。
  
  協調フィルタリング:
  「似たユーザーが好んだ記事を推薦する」
  ユーザーAとBが多くの記事に同じ「いいね」をしているなら、
  Aが読んでBが読んでいない記事をBに推薦する。
  利点: 記事の内容を分析しなくてもいい
  欠点: Cold Start (新規ユーザー/新規記事はデータがない)
  
  コンテンツベース:
  「ユーザーが好んだ記事と似た内容の記事を推薦する」
  記事の特徴量 (タグ、トピック、Embedding) で類似度を計算。
  利点: Cold Start に強い (記事の内容があればいい)
  欠点: 「いつも同じ系統」になりやすい (セレンディピティがない)
  
  Embedding とコサイン類似度:
  Embedding はテキストを高次元ベクトルに変換したものです。
  意味が似た文は近いベクトルになります。
  
  なぜユークリッド距離ではなくコサイン類似度か:
  ユークリッド距離はベクトルの「大きさ」に影響されます。
  長い文と短い文ではベクトルの大きさが異なるため、
  意味が似ていても距離が大きくなります。
  コサイン類似度は「方向」だけを見るため、
  文の長さに依存しません。
  
  HNSW (Hierarchical Navigable Small World):
  数百万のベクトルから最も近いものを見つける近似最近傍検索。
  厳密な全探索は O(n) で遅すぎるため、
  グラフ構造を使って O(log n) で近似解を見つけます。
  精度は 95%+ で、速度は数百倍速い。

----------------------------------------------------------------------
  RAG (Retrieval-Augmented Generation)
----------------------------------------------------------------------
  なぜ RAG が必要か:
  LLM は学習データにない情報を知りません (知識カットオフ)。
  また、知っている情報でも間違った回答をすることがあります (ハルシネーション)。
  
  RAG の仕組み:
  1. ユーザーの質問を受け取る
  2. 関連する文書を検索して取得 (Retrieval)
  3. 取得した文書をコンテキストとして LLM に渡す
  4. LLM がコンテキストを基に回答を生成 (Generation)
  
  Chunking (チャンキング):
  長い文書をそのまま Embedding するのではなく、
  適切な長さに分割します。
  - 固定長チャンク: 512トークンずつ分割 (重複あり)
  - セマンティックチャンク: 意味的なまとまりで分割 (段落、セクション)
  
  BM25:
  テキスト検索の古典的アルゴリズムです。
  TF-IDF の改良版で、単語の出現頻度と文書長を考慮します。
  「Kafka」で検索すると、「Kafka」が多く含まれる文書が上位に来ます。
  
  Hybrid 検索:
  BM25 (キーワードマッチ) + Dense Retrieval (意味的類似度) を組み合わせ、
  両方の長所を活かします。

----------------------------------------------------------------------
  ガードレールと安全性
----------------------------------------------------------------------
  AIをプロダクションで使うには安全策が不可欠です。
  
  Prompt Injection:
  ユーザーが悪意ある入力でAIの動作を乗っ取る攻撃。
  
  Direct Injection:
  ユーザーが直接「以前の指示を無視して...」と入力する。
  
  Indirect Injection:
  検索結果に含まれるWebページに悪意ある指示が埋め込まれている。
  RAGで取得したコンテキストに攻撃が含まれるケース。
  
  4層防御:
  Layer 1: 入力サニタイズ (特殊文字エスケープ、長さ制限)
  Layer 2: プロンプト分離 (System/User/Tool の境界を明確に)
  Layer 3: LLM ベース検出器 (入力が攻撃かどうかを分類)
  Layer 4: 出力検証 (構造化出力の強制、PII マスキング)
  
  RAGAS 評価メトリクス:
  RAGの品質を定量的に測る指標群です。
  
  Faithfulness (忠実度):
  回答がコンテキストに基づいているか (ハルシネーションしていないか)
  
  Context Precision (文脈精度):
  検索で取得したコンテキストが正確か (不要な文書が混じっていないか)
  
  Answer Relevancy (回答関連性):
  回答がユーザーの質問に答えているか

  【面接向けに整理すると】
  「RAGの品質をどう評価するか？」
  
  「RAGASフレームワークの主要メトリクスを使います。
  Faithfulnessでハルシネーションがないか確認し、
  Context Precisionで検索精度を評価し、
  Answer Relevancyで回答の適切さを測ります。
  これらを組み合わせたスコアで、チャンキング戦略や
  検索アルゴリズムの改善効果を定量的に判断します。」

  Chapter 6: コードの品質を守る (深掘り)

----------------------------------------------------------------------
  デザインパターン - なぜ使うのか
----------------------------------------------------------------------
  デザインパターンは「よくある問題に対する定型的な解決策」です。
  パターンそのものより、それが解決する「問題」を理解することが重要です。
  
  Observer パターン:
  問題: 投稿が作成されたら、フォロワーに通知したい。
  しかし Post クラスが Notification クラスを直接呼ぶと密結合になる。
  解決: Post が「投稿されたよ」とイベントを発行し、
  Notification が「投稿イベントを購読」する。
  Post は誰が聞いているか知らなくていい。
  
  Strategy パターン:
  問題: 推薦アルゴリズムを切り替えたい (協調フィルタリング/コンテンツベース)。
  if 文で分岐すると、アルゴリズム追加のたびにコードが肥大化する。
  解決: 各アルゴリズムを共通インターフェースで実装し、
  実行時に差し替え可能にする。
  
  Circuit Breaker パターン:
  問題: 外部サービス (AI API等) が障害時に、
  リクエストし続けるとタイムアウトが連鎖してシステム全体が遅くなる。
  解決: 3つの状態で制御する。
  - Closed: 通常通りリクエスト
  - Open: リクエストを即座に失敗させる (外部に送らない)
  - Half-Open: 少数のリクエストを試して回復を確認
  
  失敗が閾値を超えたら Closed → Open に遷移。
  一定時間後に Open → Half-Open に遷移して回復を試す。

----------------------------------------------------------------------
  テストピラミッド
----------------------------------------------------------------------
  テストには種類があり、バランスが重要です。
  
  テストピラミッド:
         /  E2E (10%)  \     ← 遅い、壊れやすい、高コスト
        / Integration   \   ← 中程度
       / Unit (70%)      \  ← 速い、安定、低コスト
  
  なぜ逆ピラミッド (E2Eが多い) は危険か:
  1. E2E テストは遅い (ブラウザ起動、DB接続等)
  2. 壊れやすい (UIの変更で大量のテストが壊れる)
  3. 原因特定が困難 (どこが壊れたか分かりにくい)
  4. CI が遅くなり、開発速度が低下
  
  Mock vs Real DB:
  Mock を使うべき場合:
  - 外部API (レート制限がある、課金される)
  - 単体テストで他のコンポーネントを隔離したい
  
  Real DB を使うべき場合:
  - SQL クエリの正しさを検証したい
  - トランザクションの動作を確認したい
  - ORM のマッピングが正しいか確認したい
  → Docker で使い捨てDBを起動するのが現代的なアプローチ
  
  Property-Based Testing:
  通常のテスト: 具体的な入力と期待値を書く
  Property-Based: 「どんな入力でもこの性質が成り立つ」を検証
  
  例: sorted() のテスト
  通常: assert sorted([3,1,2]) == [1,2,3]
  Property: assert sorted(sorted(xs)) == sorted(xs)  # 冪等性
  Property: assert len(sorted(xs)) == len(xs)         # 長さ保存
  
  ランダムな入力を大量に生成して性質を検証するため、
  人間が思いつかないエッジケースを見つけられます。

----------------------------------------------------------------------
  アルゴリズムの実務適用
----------------------------------------------------------------------
  面接で学ぶアルゴリズムは実務でも使います。
  
  Trie (前方一致木) → オートコンプリート:
  「Kub」と入力すると「Kubernetes」「Kubelet」が候補に出る。
  Trie は文字を1つずつノードに持つ木構造で、
  共通の接頭辞を共有します。
  「Kub」のノードまで辿れば、その下に候補が全てある。
  
  Heap (ヒープ) → Top-K 問題:
  「トレンドの投稿トップ10」を求める場合、
  全投稿をソートすると O(n log n) かかります。
  サイズ K の Min-Heap を使うと O(n log K) で済みます。
  n = 100万, K = 10 なら、計算量が約5倍速くなります。
  
  Sliding Window → リアルタイム集計:
  「過去5分間のいいね数」を常に最新で保つ場合、
  タイムスタンプ付きのデキュー (両端キュー) を使い、
  古いイベントを左から削除、新しいイベントを右に追加します。
  ウィンドウ内の合計を O(1) で更新できます。

  【面接向けに整理すると】
  「テスト戦略をどう設計するか？」
  
  「テストピラミッドに基づき、Unit 70%、Integration 20%、E2E 10% の
  バランスを目指します。Unitテストは高速で安定しており、
  CIパイプラインで最初に実行して即座にフィードバックを得ます。
  Integration テストではDBとの接続やAPI間の連携を検証し、
  E2Eは主要なユーザーフローのみカバーします。
  外部APIはMockし、DBはDockerで使い捨てインスタンスを使います。」

======================================================================
  Chapter 7: コンテナに詰めて出荷する (深掘り)
======================================================================

----------------------------------------------------------------------
  Docker とコンテナの本質
----------------------------------------------------------------------
  Docker とは:
  アプリケーションとその依存関係を「コンテナ」にパッケージングするツールです。
  「自分のマシンでは動くのに、本番では動かない」問題を解決します。
  
  コンテナ vs VM:
  VM (仮想マシン):
  - ハイパーバイザーがハードウェアを仮想化
  - 各VMにゲストOSが必要 (数GB)
  - 起動: 数十秒〜分
  - 完全な隔離 (カーネルも別)
  
  コンテナ:
  - ホストOSのカーネルを共有
  - OS ライブラリだけパッケージ (数十MB)
  - 起動: ミリ秒〜秒
  - 論理的な隔離 (namespaces + cgroups)
  
  namespaces: 何を隔離するか
  - PID namespace: プロセスID空間を分離 (コンテナ内はPID 1から開始)
  - NET namespace: ネットワークスタックを分離 (独自のIPアドレス)
  - MNT namespace: ファイルシステムを分離 (独自のルートディレクトリ)
  - UTS namespace: ホスト名を分離
  - IPC namespace: プロセス間通信を分離
  - USER namespace: ユーザーIDを分離 (rootless コンテナ)
  
  cgroups: リソースを制限する
  - CPU: cpu.cfs_quota_us で CPU 時間の上限を設定
  - Memory: memory.limit_in_bytes でメモリ上限
  - I/O: blkio で読み書き帯域制限
  
  UnionFS (OverlayFS):
  Docker イメージはレイヤーの積み重ねです。
  base image (Ubuntu) + アプリ + 設定 が別々のレイヤーとして存在し、
  OverlayFS が1つのファイルシステムに見せます。
  共通レイヤーは複数コンテナで共有されるため、ストレージ効率が良い。

----------------------------------------------------------------------
  Kubernetes
----------------------------------------------------------------------
  Kubernetes はコンテナのオーケストレーションプラットフォームです。
  複数のコンテナのデプロイ、スケーリング、管理を自動化します。
  
  リソース階層:
  
  Pod:
  - 1つ以上のコンテナの最小実行単位
  - 同一Pod内のコンテナはネットワークとストレージを共有
  - 通常は1Pod = 1コンテナ
  
  Deployment:
  - Podのレプリカ数を宣言的に管理
  - 例: replicas: 3 → 常に3つのPodを維持
  - ローリングアップデート: 新バージョンを段階的に入れ替え
  
  Service:
  - Pod群への安定したネットワークエンドポイント
  - Podは生まれては消えるがServiceのIPは変わらない
  - ロードバランシング機能を持つ
  
  Ingress:
  - 外部トラフィックをServiceにルーティング
  - パスベース: /api → API Service, /web → Frontend Service
  - TLS終端もここで行える
  
  HPA (Horizontal Pod Autoscaler):
  CPU使用率やカスタムメトリクスに基づいてPod数を自動調整。
  CPU 80% → Pod を増やす、30% → Pod を減らす。

----------------------------------------------------------------------
  CI/CD パイプライン
----------------------------------------------------------------------
  コードの変更を安全に本番環境に届ける仕組みです。
  
  パイプラインの各ステージ:
  
  Lint: コードスタイルの検査 (flake8, eslint)
  → なぜ最初か: 最も速く完了し、明らかな問題を早期に検出
  
  Test: ユニットテスト + インテグレーションテスト
  → 機能が壊れていないことを確認
  
  Build: Docker イメージのビルド
  → マルチステージビルドで最終イメージを小さく
  
  Push: イメージをレジストリに登録
  
  Deploy: 本番環境にデプロイ
  
  Verify: Smoke テスト、ヘルスチェック
  
  デプロイ戦略:
  
  Rolling Update:
  - 古い Pod を1つずつ新しいバージョンに置き換える
  - ダウンタイムなし
  - 問題: 新旧が混在する期間がある
  
  Blue-Green:
  - 新環境 (Green) を丸ごと用意し、切り替える
  - 問題発生時は旧環境 (Blue) に即座に戻せる
  - コスト: 環境が2倍必要
  
  Canary:
  - まず5%のトラフィックだけ新バージョンに流す
  - メトリクスを確認して問題なければ 25% → 100% と拡大
  - 最もリスクが低い方式

  Chapter 8: クラウドに載せる (深掘り)
======================================================================

----------------------------------------------------------------------
  コンピュートの抽象度
----------------------------------------------------------------------
  クラウドのコンピュートサービスは「抽象度」で整理できます:
  
  低い ← 抽象度 → 高い
  EC2 → ECS → Fargate → App Runner → Lambda
  
  EC2: 仮想サーバーそのもの。OSから自由に設定。運用負荷は最大。
  ECS: コンテナのオーケストレーション。EC2上でコンテナを管理。
  Fargate: サーバーレスコンテナ。EC2の管理が不要。
  App Runner: ソースコードやイメージからワンクリックデプロイ。
  Lambda: 関数単位の実行。イベント駆動。最大15分。
  
  選択基準:
  - GPU が必要 → EC2 (GPU インスタンス)
  - 長時間バッチ → EC2 or ECS
  - Web API → Fargate or App Runner
  - イベント処理 → Lambda
  - プロトタイプ → App Runner (最速デプロイ)
  
  抽象度が高いほど:
  - 運用コストが下がる (パッチ適用、スケーリング等をクラウドが行う)
  - 制御が減る (OS設定、カーネルチューニング等ができない)

----------------------------------------------------------------------
  Well-Architected Framework
----------------------------------------------------------------------
  AWS が提唱する設計の指針。6本の柱で構成:
  
  1. 運用の優秀性: 自動化、モニタリング、継続改善
  2. セキュリティ: 最小権限、暗号化、監査ログ
  3. 信頼性: 障害からの自動回復、スケーリング
  4. パフォーマンス効率: 適切なリソースタイプの選択
  5. コスト最適化: 不要リソースの削除、RI/Spot の活用
  6. 持続可能性: エネルギー効率、リソース最適化
  
  TechPulse を各柱で検証する例:
  - セキュリティ: API Gateway で認証集約、VPC で分離、KMS で暗号化
  - 信頼性: マルチAZ デプロイ、Auto Scaling、Circuit Breaker
  - コスト: 開発環境は夜間シャットダウン、本番は RI + Spot 混合

----------------------------------------------------------------------
  FinOps
----------------------------------------------------------------------
  クラウドのコストを管理・最適化する実践です。
  
  料金体系の理解:
  On-Demand: 使った分だけ。最も高い。
  Reserved Instance (RI): 1年/3年契約で最大75%割引。予測可能なワークロード向け。
  Spot Instance: 余剰キャパシティ。最大90%割引。中断される可能性あり。
  
  使い分け:
  - ベースライン (常に必要な量) → RI
  - ピーク (一時的に必要な量) → On-Demand + Auto Scaling
  - バッチ処理 (中断許容) → Spot
  
  コスト削減の実践:
  1. 不要なリソースの削除 (使っていない EBS、未使用の Elastic IP)
  2. 適切なインスタンスサイズ (過剰スペックの是正)
  3. 開発環境の夜間停止
  4. S3 ライフサイクルポリシー (古いデータを低コスト層に移動)

======================================================================
  Chapter 9: 本番を守る - SRE/可観測性 (深掘り)
======================================================================

----------------------------------------------------------------------
  SLI / SLO / SLA
----------------------------------------------------------------------
  SLI (Service Level Indicator):
  サービスの健全性を測る「指標そのもの」。
  例: リクエスト成功率、レイテンシ p99
  
  SLO (Service Level Objective):
  SLIの「目標値」。内部的に設定する。
  例: リクエスト成功率 99.9%、p99 レイテンシ < 500ms
  
  SLA (Service Level Agreement):
  顧客との「契約」。SLO を下回った場合のペナルティ (返金等) を含む。
  例: 月間可用性 99.9% を保証。未達成時は利用料の10%返金。
  
  SLO は SLA より厳しく設定すべきです。
  SLA 違反でペナルティが発生する前に、SLO 違反で内部アラートを出す。

----------------------------------------------------------------------
  Error Budget
----------------------------------------------------------------------
  Error Budget (エラーバジェット) とは:
  「SLOで許容される障害の量」です。
  
  SLO が 99.9% の場合:
  月間 (30日): 0.1% の障害が許容 = 43.2 分
  これが Error Budget です。
  
  使い方:
  Error Budget が残っている → 新機能のリリースを積極的に行う
  Error Budget を使い切った → リリースを止めて安定性に投資する
  
  これにより、開発チームと運用チームの対立を
  データに基づいて解消できます。

----------------------------------------------------------------------
  監視の 3 本柱
----------------------------------------------------------------------
  Metrics (メトリクス):
  数値データの時系列。CPU使用率、リクエスト数、エラー率。
  「何が起きているか」の全体像を把握する。
  ツール: Prometheus + Grafana
  
  Logs (ログ):
  イベントの詳細な記録。エラーメッセージ、スタックトレース。
  「なぜ起きたか」の詳細を調査する。
  ツール: ELK (Elasticsearch + Logstash + Kibana)
  
  Traces (トレース):
  リクエストがサービス間をどう流れたかの記録。
  「どこで遅くなったか」を特定する。
  ツール: Jaeger, Zipkin, AWS X-Ray
  
  OpenTelemetry:
  Metrics, Logs, Traces を統一的に収集するフレームワーク。
  ベンダー中立で、どの監視ツールにもデータを送れます。

----------------------------------------------------------------------
  インシデント対応
----------------------------------------------------------------------
  OODA ループ:
  Observe (観察): アラートの内容、影響範囲を確認
  Orient (状況判断): 過去の類似事例、変更履歴を確認
  Decide (決断): 対応方針を決定 (ロールバック？ スケールアウト？)
  Act (実行): 対応を実施し、結果を確認
  
  ポストモーテム:
  障害後に行う振り返りです。
  
  Blameless (非難しない) が原則:
  「誰がミスしたか」ではなく「なぜシステムがミスを防げなかったか」を問う。
  人を責めると、次から問題を隠すようになり、組織が学ばなくなる。
  
  ポストモーテムに含めるもの:
  1. 何が起きたか (タイムライン)
  2. 影響範囲
  3. 根本原因
  4. 再発防止策 (アクションアイテム)

----------------------------------------------------------------------
  p99 テールレイテンシ
----------------------------------------------------------------------
  なぜ平均ではダメか:
  平均レイテンシが 50ms でも、1%のリクエストが 5000ms かかっている場合、
  ユーザー体験は悪い。
  
  p99 = 99パーセンタイル:
  「100回のリクエストのうち99回はこの時間以内に完了する」
  つまり、最も遅い 1% の体験を表します。
  
  大規模サービスでは p99.9 も重要:
  ユーザーが1回のページ表示で 100 回のAPIを呼ぶ場合、
  1回でも遅いとページ全体が遅くなる。
  p99 が 100ms なら、100回呼ぶとほぼ確実に1回は 100ms 超が発生する。

----------------------------------------------------------------------
  USE法 と RED法
----------------------------------------------------------------------
  USE法 (リソース視点):
  U: Utilization (使用率) - CPU 80% なら 80%
  S: Saturation (飽和度) - キューにどれだけ溜まっているか
  E: Errors (エラー) - ハードウェアエラー、ドライバエラー
  
  各リソース (CPU, メモリ, ディスク, ネットワーク) に対して
  U, S, E を確認する。ボトルネック特定に有効。
  
  RED法 (サービス視点):
  R: Rate (処理率) - 1秒あたりのリクエスト数
  E: Errors (エラー率) - 失敗したリクエストの割合
  D: Duration (所要時間) - レイテンシの分布
  
  マイクロサービスの健全性を測るのに有効。

======================================================================
  Chapter 10: セキュリティを固める (深掘り)
======================================================================

----------------------------------------------------------------------
  SQL Injection の仕組み
----------------------------------------------------------------------
  SQL Injection はなぜ起きるか:
  プログラムが「SQL命令」と「ユーザーデータ」を区別できないからです。
  
  文字列結合で SQL を組み立てると、
  ユーザー入力が SQL の一部として解釈される可能性があります。
  
  なぜ Prepared Statement で防げるか:
  Prepared Statement では SQL の構文解析が先に行われます。
  その後にデータがバインドされるため、
  データはあくまで「値」として扱われ、
  SQL の構文を変更することはできません。
  
  XSS (Cross-Site Scripting):
  攻撃者が悪意あるスクリプトをWebページに埋め込む。
  他のユーザーがそのページを開くとスクリプトが実行される。
  防御: 出力時のHTMLエスケープ、Content-Security-Policy ヘッダ
  
  CSRF (Cross-Site Request Forgery):
  攻撃者が被害者のブラウザに「意図しないリクエスト」を送らせる。
  被害者がログイン中のサービスに対して、知らない間にリクエストが送られる。
  防御: CSRF トークン、SameSite Cookie
  
  XSS vs CSRF の違い:
  XSS: 攻撃者のスクリプトが被害者のブラウザで実行される
  CSRF: 被害者のブラウザが攻撃者の意図したリクエストを送る

----------------------------------------------------------------------
  暗号化の全レイヤー
----------------------------------------------------------------------
  At rest (保存時):
  ディスクに保存されたデータの暗号化。
  AES-256 が標準。鍵は KMS (Key Management Service) で管理。
  
  In transit (通信時):
  ネットワーク上を流れるデータの暗号化。
  TLS 1.3 (ECDHE + AES-GCM)。
  
  ゼロトラスト:
  従来の「境界防御」: ファイアウォールの中は信頼する
  ゼロトラスト: 誰も信頼しない。全アクセスを検証する。
  
  原則:
  1. 明示的に検証 (Entra ID で全リクエストを認証)
  2. 最小権限 (必要最小限の権限だけ付与)
  3. 侵害を想定 (侵入されることを前提に設計)

----------------------------------------------------------------------
  A/B テスト設計
----------------------------------------------------------------------
  A/Bテストとは:
  ユーザーを2群に分け、異なるバージョンを見せて
  どちらが優れているかをデータで判断する手法。
  
  p値とは:
  「帰無仮説 (差がない) が正しい場合に、
  今回のデータ以上に極端な結果が偶然生じる確率」
  
  よくある誤解: p値は「差がある確率」ではない
  正確には: p値は「差がない前提での、偶然の確率」
  
  p < 0.05 なら「偶然とは考えにくい」として帰無仮説を棄却する。
  ただし 0.05 という閾値自体に絶対的な根拠はなく、慣習です。
  
  多重比較問題:
  A/B/C の3群で比較すると、3つの検定を行う。
  各検定で5%の偽陽性リスクがあるなら、
  少なくとも1回偽陽性が出る確率は 1-(0.95^3) ≈ 14.3%。
  Bonferroni 補正: 有意水準を比較回数で割る (0.05/3 ≈ 0.017)。

----------------------------------------------------------------------
  勾配降下法
----------------------------------------------------------------------
  機械学習モデルの学習は「パラメータの最適化」です。
  損失関数 (予測と正解のズレ) を最小化するパラメータを見つける。
  
  勾配降下法:
  1. 現在のパラメータで損失を計算
  2. 損失の勾配 (パラメータを変えたら損失がどう変わるか) を計算
  3. 勾配の逆方向にパラメータを更新 (損失が減る方向に進む)
  4. 1-3 を繰り返す
  
  学習率 (Learning Rate):
  パラメータの更新幅を制御する。
  大きすぎると最適値を飛び越える。
  小さすぎると収束が遅い。
  実務ではスケジューラで徐々に小さくすることが多い。

----------------------------------------------------------------------
  LoRA と RLHF
----------------------------------------------------------------------
  LoRA (Low-Rank Adaptation):
  LLM を特定ドメインに適応させたいが、
  全パラメータの再訓練はコスト大。
  
  解決: 元の重み行列 W に低ランク行列 ΔW = A × B を加算。
  A は d×r、B は r×d (r << d、例: r=8)。
  訓練パラメータ数が桁違いに少なくなる。
  
  直感: 元のモデルは「凍結」し、
  小さな「アダプタ」だけ学習させる。
  フルファインチューニング: GPU 8台 → LoRA: GPU 1台
  
  RLHF (Reinforcement Learning from Human Feedback):
  1. SFT: 良い例で教師あり学習
  2. Reward Model: 人間が「この回答は良い/悪い」と評価
  3. PPO: Reward Model のスコアを最大化するように LLM を更新
  
  PPO (Proximal Policy Optimization):
  強化学習のアルゴリズムで、
  ポリシー (どう応答するかの戦略) を少しずつ改善する。
  大きく変更しすぎないよう制約をかけるのが「Proximal」の意味。

----------------------------------------------------------------------
  RICE スコア
----------------------------------------------------------------------
  機能の優先順位を定量的に決めるフレームワーク。
  
  R (Reach): この機能が影響するユーザー数
  I (Impact): 各ユーザーへの影響度 (3=大, 2=中, 1=小, 0.5=極小)
  C (Confidence): 推定の確信度 (100%=高, 80%=中, 50%=低)
  E (Effort): 開発工数 (人月)
  
  RICE Score = (Reach × Impact × Confidence) / Effort
  
  例:
  AI推薦機能: Reach=10000, Impact=2, Confidence=80%, Effort=3
  → (10000 × 2 × 0.8) / 3 = 5333
  
  ダークモード: Reach=50000, Impact=0.5, Confidence=100%, Effort=0.5
  → (50000 × 0.5 × 1.0) / 0.5 = 50000
  
  → ダークモードのほうがスコアが高い！
  直感に反するかもしれないが、工数に対するリーチが大きい。

----------------------------------------------------------------------
  DORA メトリクス
----------------------------------------------------------------------
  開発組織のパフォーマンスを測る4つの指標:
  
  1. Deployment Frequency (デプロイ頻度):
     どれくらい頻繁に本番にデプロイするか
     Elite: 日に複数回 / Low: 月1回未満
  
  2. Lead Time for Changes (変更リードタイム):
     コミットから本番デプロイまでの時間
     Elite: 1日未満 / Low: 6ヶ月超
  
  3. Mean Time to Recovery (MTTR):
     障害発生から回復までの時間
     Elite: 1時間未満 / Low: 6ヶ月超
  
  4. Change Failure Rate (変更失敗率):
     デプロイのうち障害を引き起こした割合
     Elite: 0-15% / Low: 46-60%
  
  重要な洞察:
  速度と安定性はトレードオフではない。
  Elite チームは「速く」かつ「安定して」デプロイしている。
  CI/CD、自動テスト、Feature Flag が鍵。

----------------------------------------------------------------------
  Conway's Law
----------------------------------------------------------------------
  「組織のコミュニケーション構造が、
  システムのアーキテクチャに反映される」
  
  例: フロントエンドチーム、バックエンドチーム、
  DBチームが別々なら、システムも3層に分かれやすい。
  
  逆コンウェイ:
  望ましいアーキテクチャから逆算して組織を設計する。
  マイクロサービスにしたいなら、
  サービスごとにクロスファンクショナルなチームを作る。
